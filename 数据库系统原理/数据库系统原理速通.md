# 第一章：绪论
1. **⭐数据库的4个基本概念**
   1. 数据 data：描述事物的符号，数据库中存储的基本对象
   2. 数据库 DataBase，**DB**：**长期存储在计算机内的、有组织、可共享的大量数据的集合**
   3. 数据库管理系统 DataBase Management System，**DBMS**：位于**用户与操作系统**之间的一层数据管理软件
   4. 数据库系统 DataBase System，**DBS**：由**数据库、数据库管理系统、应用程序、数据管理员(DBA)** 组成。
2. **⭐数据库系统的特点**
   1. 数据结构化
   2. 数据的共享性高，冗余度低并且容易扩充
   3. 数据独立性高
      1. **物理独立性**：应用程序与物理存储相互独立，数据的物理存储改变，应用程序不改变
      2. **逻辑独立性**：应用程序与逻辑结构相互独立，数据逻辑结构改变，应用程序不改变
   4. 数据由数据库管理系统统一管理和控制
3. **⭐数据模型**
   1. 分类
      1. 概念模型（信息模型）
         1. 典型代表：E-R 模型（实体-联系模型）
      2. 逻辑模型
         1. 用于描述数据库表结构、数据关系、约束
         2. 分类：层次模型（树形结构，1对多）、网状模型（图结构，多对多）、**关系模型**（二维表）
      3. 物理模型
         1. 描述数据在磁盘上的物理存储结构，是对数据最底层的抽象
4. **⭐数据模型的组成三大要素**
   1. **数据结构**：描述系统**静态特性**$\to$描述数据库的组成对象以及对象之间的联系
   2. **数据操作**：描述系统**动态特性**$\to$即对数据库中各种对象的实例允许的操作的集合，其类型有查询和更新（增删改）
   3. **数据的完整性约束条件**：用以限定符合数据模型的数据库状态以及状态的变化，以保证数据的正确、有效与相容，完整性约束条件是一组完整性规则的整合
5. **⭐数据库系统的三级范式**
   1. **外模式（子模式/用户模式）**：数据库用户能看见和使用的局部数据的逻辑结构和特征的描述，数据库用户的数据视图，是与某应用有关的数据的逻辑表示
      ![[Pasted image 20260622155725.png]]
   2. **模式（逻辑模式）**：数据库中全体数据的逻辑结构和特征描述，是所有用户的公共数据视图
   3. **内模式（存储模式）**：数据物理结构和存储方法的描述，是数据在数据库内部的组织方式
6. **⭐数据库的二级映像与数据独立性**
   1. **外模式/模式映像**：保证了数据的逻辑独立性
      1. 当模式改变时，对外模式/模式的映像做出改变，保证外模式不变，应用程序由外模式编写，从而应用程序不变
      2. **举例**：数据库改表结构加字段，外模式不变的情况下，应用程序不用改。
   2. **模式/内模式映像**：保证了数据的物理独立性
      1. 当数据库的存储结构改变时，对模式/内模式映像做出改变，使模式不变，进而应用程序不变
      2. **举例**：换存储引擎，应用程序无需更改。
7. **⭐区分 C/S、B/S**

   | 架构 | 全称 | 客户端 | 特点 |
   | --- | --- | --- | --- |
   | `C/S` | Client/Server（客户机/服务器） | 需要安装专用客户端 | 客户端承担部分业务逻辑，交互能力强，维护升级成本较高 |
   | `B/S` | Browser/Server（浏览器/服务器） | 通过浏览器访问 | 主要逻辑在服务器端，部署维护方便，跨平台性好 |

   **速记**：`C/S` 看“客户端软件”，`B/S` 看“浏览器访问”。

8. **⭐DBA 的职责**
   1. 决定数据库中的信息内容和结构
   2. 决定数据库的存储结构和存取策略
   3. 定义数据的安全性要求和完整性约束条件
   4. 监控数据库运行，负责备份、恢复与性能改进

# 第四章：数据库安全性（授权）
1. **不安全因素**
   1. 非授权用户对数据库的恶意存取和破坏
   2. 数据库中重要数据的泄露
   3. 安全环境的脆弱性
2. **⭐数据库安全性控制**
   1. 用户身份鉴别：静态口令鉴别、动态口令鉴别、生态特征鉴别、智能卡鉴别
   2. 存取控制
   3. 自主存取控制方法
3. **为什么要授权**
   1. 授权是指授予（GRANT）和收回（REVOKE），是数据库安全性控制中的自主存取控制方法，是为了保护数据库，防止不合法使用所造成的数据泄露、更改或破坏
4. **⭐如何授权：授予 GRANT**
   发出如下sql语句的可以是：数据库管理员、数据库对象创建者、已经拥有该权限的用户
      ```sql
      GRANT 权限 ON 对象类型 对象名 TO 用户名 [WITH GRANT OPTION]
      ```
   1. 字段解析
      1. **权限**：查询权限 `SELECT`，全部操作权限 `ALL PRIVILEGES`
      2. **对象类型&对象名**：对象类型可以是 `TABLE` 也可以是 `VIEW`，对象名位对应的表名或者视图名
      3. **用户名**：可以是指定用户 `用户名`，也可以是全体用户 `PUBLIC`
      4. **WITH GRANT OPTION**：不存在这句的话，获得权限的用户只能使用该权限，不能传播该权限
         注意点：`SQL` 不允许循环授权，即被授权者不能把权限再收回给授权者或者其祖先
   2. 举例：把学生表的查询权限授给用户 U1
      1. `GRANT SELECT ON TABLE Student TO U1`
5. **收回授权：收回 REVOKE**
   1. 格式
      ```sql
      REVOKE 权限 ON 对象类型 对象名 FROM 用户名 [CASCADE|RESTRICT]
      ```
      2. 字段解析
         1. **CASCADE**：级联回收，将用户传播出去的权限一并收回
         2. **RESTRICT**：受限回收，若用户传播过该权限，回收将会失败
   3. 举例：将用户 U4 的修改学生学号权限收回
      1. `REVOKE UPDATE(Sno) ON TABLE Student from U4`
6. **创建数据库模式的权限**
    对创建数据库模式一类的数据库对象的授权在数据库管理员创建用户时实现。
      只有系统的**超级用户**才有权创建一个新的数据库用户，新创建的数据库用户有三种权限：DBA、RESOURCE、CONNECT（默认）
      ```sql
      CREATE USER username [WITH DBA|RESOURCE|CONNECT]
      ```
    **字段详解**：DBA什么都行，RESOUCE可以创建表/视图、把权限授予其他用户，CONNECT只能登录，依据DBA/RESOUCE给的授权进行操作
         1. **DBA**：可以创建新用户、创建模式、创建基本表和视图等，拥有对所有数据库对象的存取权限，还可以把这些权限授予一般用户
         2. **RESOUCE**：可以创建基本表和视图，但不能创建模式和新用户。数据库对象的属主可以用GRANT语句把该对象上的存取权限授予其他用户
         3. **CONNECT**：只能登陆数据库，根据其他用户或者数据库管理员授予的权限的情况对数据库对象进行范围内的操作。
7. **⭐数据库角色**
   角色是权限的集合，可以为一组拥有相同权限的用户创建一个角色，使用角色来管理数据库权限，简化授权过程
   1. **创建角色**：
      1. ```sql
         CREATE ROLE 角色名
         ```
   2. **在一个角色中添加角色/用户**：
      1. ```sql
         GRANT 角色 TO 某角色/某用户 [WITH ADMIN OPTION]
          ```
   3. **给角色授权**：
      1. ```sql
         GRANT 权限 ON 对象类型 对象名 TO 角色
         ```
   4. **角色权限收回**：
      1. ```sql
         REVOKE 权限 ON 对象类型 对象名 FROM 角色
         ```
8. **⭐视图机制**
   为不同的用户定义不同的视图，把不需要的数据隐藏，避免用户误操作
9. **审计**
   把对数据库的所有操作都记录到审计日志中，然后就可以通过日志审查这个里面是否有一些非法的行为
10. **数据加密**
   通过一些加密算法，把明文变为密文，这样别人无法查看
# 第五章：数据库完整性
1. **数据库的完整性是指数据得正确性和相容性**
   正确性：符合现实世界语义
   相容性：同一对象在不同表中的数据符合逻辑
2. **维护完整性需要实现的功能**
   1. 提供定义完整性约束条件的机制
   2. 提供完整性检查的方法
   3. 进行违约处理
3. **⭐三大完整性**
   1. **实体完整性**：主码唯一且非空
      **检查与违约处理**：检查主码是否唯一，如果不唯一则拒绝插入或修改；检查主码各个属性是否为空，只要有一个为空就拒绝插入或修改
   2. **参照完整性**：外码的约束
      检查：对被参照表和参照表进行增删改操作时可能会破坏参照完整性，必须检查以保证两个表的相容性。
      违约处理：外码约束可以在创建参照表时说明不同的违约策略
      **可能破坏参照完整性的情况及违约处理**

      | 被参照表（例如Student） | 参照表（例如SC） | 违约处理 |
      | --- | --- | --- |
      | 可能破坏参照完整性 | 插入元组 | 拒绝 |
      | 可能破坏参照完整性 | 修改外码值 | 拒绝 |
      | 删除元组 | 可能破坏参照完整性 | 拒绝/级联删除/设置为空值 |
      | 修改主码值 | 可能破坏参照完整性 | 拒绝/级联修改/设置为空值 |
   3. **用户定义完整性**：属性上约束条件的定义
      1. 属性上的约束条件
         非空 NOT NULL
         列值唯一 UNIQUE
         检查列值是否满足一个表达式 CHECK
         检查和违约处理：当向表中1插入元组或者修改属性值时，关系数据库管理系统将检查属性上的约束条件是否被满足
      2. 元组上的约束条件
         在`CREATE TABLE`语句中可以用 `CHECK` 短语定义元组上的约束条件，元组级的限制可以设置不同属性之间的取值的相互约束条件
         检查和违约处理：当向表中插入元组或者修改属性值时，DBMS会检查元组上的约束条件是否被满足，不满足则拒绝操作
4. **⭐SQL Server 中各类完整性实现的手段**

   | 完整性类型 | SQL Server 实现手段 | 重点说明 |
   | --- | --- | --- |
   | **实体完整性** | `PRIMARY KEY`、`UNIQUE`、`NOT NULL` | 主码必须**唯一且非空**；`UNIQUE` 类似候选码，用来保证某列或某组列不重复；一个表只能有一个主键，但可以有多个 `UNIQUE` 约束 |
   | **参照完整性** | `FOREIGN KEY ... REFERENCES ...`，配合 `ON DELETE`、`ON UPDATE` | 外码取值要么为空，要么来自被参照表的主码/唯一键；违约处理可选拒绝、级联、置空、置默认值 |
   | **用户定义完整性** | 数据类型、`NOT NULL`、`DEFAULT`、`CHECK`、命名约束 `CONSTRAINT`、触发器 | 用来表达具体业务规则；简单规则用 `CHECK` 等声明式约束，复杂跨表规则可用触发器 |

   **参照完整性的违约处理策略**：

   | 策略 | SQL Server / SQL 写法 | 含义 |
   | --- | --- | --- |
   | 限制 / 拒绝 | `NO ACTION`（常见默认）/ `RESTRICT` | 不允许破坏参照关系，拒绝本次更新 |
   | 级联 | `CASCADE` | 父表删除或修改时，子表对应行也自动删除或修改 |
   | 置空 | `SET NULL` | 父表变化后，子表外码置为空值 |
   | 置默认值 | `SET DEFAULT` | 父表变化后，子表外码置为默认值 |

   **简单示例**：

   ```sql
   CREATE TABLE Employee (
       Eno  CHAR(4) PRIMARY KEY,               -- 实体完整性
       Name VARCHAR(8) NOT NULL,
       Sex  CHAR(2) CHECK (Sex IN ('男', '女')), -- 用户定义完整性
       Age  INT CHECK (Age >= 18 AND Age <= 65),
       Dno  CHAR(2),
       CONSTRAINT FK_Employee_Department
           FOREIGN KEY (Dno) REFERENCES Department(Dno)
           ON DELETE NO ACTION
           ON UPDATE CASCADE                   -- 参照完整性
   );
   ```

   **补充**：教材还强调，为了后续删除或修改约束，最好用 `CONSTRAINT` 给约束命名；修改约束通常是**先 `DROP CONSTRAINT`，再 `ADD CONSTRAINT`**。

   **速记**：主键保实体，外键保参照，`CHECK/DEFAULT/NOT NULL` 管业务取值，触发器处理复杂规则。
5. **⭐触发器的基本概念**
   触发器（Trigger）是由数据库操作事件驱动的特殊过程。用户定义触发器后，任何用户对指定表或视图执行 `INSERT`、`DELETE`、`UPDATE` 时，DBMS 会**自动激活**相应触发动作，在数据库服务器端进行集中的完整性控制。

   1. **触发器由两部分组成**
      1. **触发事件**：说明什么操作会触发，如 `INSERT`、`DELETE`、`UPDATE`；其中 `UPDATE` 还可以指定某些列被修改时触发。
      2. **触发动作**：触发后自动执行的一组 SQL 语句。

   2. **按触发时间分类**

      | 类型 | 含义 |
      | --- | --- |
      | `BEFORE` | 事件发生前触发（标准 SQL 中有，SQL Server 的 DML 触发器不使用这种写法） |
      | `AFTER` / `FOR` | 事件成功执行后触发；SQL Server 中 `FOR` 基本等同于 `AFTER` |
      | `INSTEAD OF` | 不执行原来的增删改语句，而是改为执行触发器中的动作，常用于视图或复杂更新逻辑 |

   3. **SQL Server 中的两个临时逻辑表**

      | 操作 | `INSERTED` 表 | `DELETED` 表 |
      | --- | --- | --- |
      | `INSERT` | 保存新插入的记录 | 空 |
      | `DELETE` | 空 | 保存被删除的旧记录 |
      | `UPDATE` | 保存修改后的新记录 | 保存修改前的旧记录 |

      **速记**：`INSERTED` 存新值，`DELETED` 存旧值；`UPDATE` 相当于“删旧 + 插新”，所以两张表都有。

   4. **SQL Server 创建触发器的基本格式**

      ```sql
      CREATE TRIGGER 触发器名 ON 表名或视图名
      {FOR | AFTER | INSTEAD OF} {[DELETE] [, INSERT] [, UPDATE]}
      AS
          -- 触发动作
      ```

      如果只想在某列被插入或修改时执行，可用：

      ```sql
      IF UPDATE(列名)
          -- 触发动作
      ```

      注意：`IF UPDATE(列名)` 用于 `INSERT` 或 `UPDATE`，不能用于 `DELETE`。

   5. **作用**
      1. 实现 `PRIMARY KEY`、`FOREIGN KEY`、`CHECK` 难以表达的复杂完整性约束。
      2. 自动维护冗余数据或派生数据，例如同步更新工资表、统计表。
      3. 记录审计日志，例如删除职工时把旧记录保存到历史表。
      4. 实现更复杂的级联操作或阻止非法操作。

   **考试答题模板**：触发器是由 `INSERT/DELETE/UPDATE` 等数据库操作事件自动激活的特殊过程，由触发事件和触发动作组成；SQL Server 常用 `AFTER/FOR` 和 `INSTEAD OF` 触发器，并通过 `INSERTED`、`DELETED` 临时表访问更新前后的数据。

# 第七章：数据库设计的步骤
1. **需求分析**
   调查机构情况与熟悉业务活动，明确用户的需求，确定系统的边界，生成数据字典和用户需求规格说明书
2. **概念结构设计**
   将需求分析得到的用户需求抽象为概念模型，绘制 `E-R` 图
3. **逻辑结构设计**
   将 `E-R` 图转换为与 `DBMS` 相符合的逻辑结构（包括数据库模式和外模式），例如 `E-R`图向关系模型转换，再根据规范化理论对数据模型进行优化，设计用户子模式
4. **物理结构设计**
   通常关系数据库物理设计的内容包含关系模式选择存取方法，以及设计关系、索引等数据库文件的物理存储结构
5. **数据库实施**
   建立实际数据库建构、试运行、装入数据
6. **数据库运行和维护**
   维护数据库的安全性、完整性控制以及系统的转储和恢复；性能的监督、分析和改进；增加新功能；发现错误和修改错误
# 第八章：数据库编程
1. **⭐嵌入式SQL与主语言之间的通信**
   **备注**：只考名词解释部分。
   SQL是描述性的面向集合的语句，负责操作数据库；高级语言语句是过程性的面向记录的语句，负责控制程序流程通信。
   在嵌入式SQL中，主语言为C语言时，语法格式为： `EXEC SQL <SQL语句>`
   Java的语法格式： `#SQL {<SQL语句>}`
   嵌入式SQL与主语言之间的通信包括：
   1. SQL向主语言传递执行状态信息，主要用SQL通信区实现
   2. 主语言向SQL提供参数，主要用主变量实现
   3. 将SQL语句查询数据库的结构交给诸语言处理，主要用主变量和游标实现
   
   **名词解释**：
   1. **SQL通信区**
      SQL 语句执行后的执行状态信息将送到 SQL 通信区中，应用程序从中取出这些信息，据此决定接下来执行的语句。
      SQL 通信区在应用程序中用 `EXEC SQL INCLUDE SQLCA` 加以定义
   2. **主变量**
      SQL 语句中使用的主语言程序变量简称主变量（即 SQL 使用主语言的变量）
      输入主变量由应用程序赋值，SQL 引用；输出主变量由 SQL 赋值或设置状态信息，返回应用程序
      一个主变量可以附带一个任选的指令变量（整型），用来指示主变量的值或条件
      主变量名和指示变量名前面要加冒号（:）作为标志
      所有主变量和指示变量必须在 SQL 语句 `BGEIN DECLARE SECTION 与 END DECLARE SECTION` 之间进行说明
   3. **游标**
      SQL 是面向集合的，一条 SQL 语句可以产生或处理多条记录，而主语言时面向记录的，一组变量一次只能存放一条记录，因此引入游标来协调这两种不同的处理方式
      一个数据缓冲区，用来存放 SQL 语句的执行结果，每个游标区都有自己的名字
   4. **建立连接**
      嵌入式 SQL 访问数据库必须先连接数据库。
      建立连接的嵌入式 SQL 语句为：
      ```sql
      EXEC SQL CONNECT TO target [AS connection-name] [USER user-name];
      ```
      其中 `target` 是要连接的数据库服务器，`connection-name` 是可选的连接名。
      运行过程中若要修改当前连接，对应的嵌入式 SQL 语句为：
      ```sql
      EXEC SQL SET CONNECTION connection-name|DEFAULT;
      ```
   5. **关闭连接**
      对应的嵌入式 SQL 语句为：
      ```sql
      EXEC SQL DISCONNECT [connection];
      ```
# 第十章：数据库恢复技术
1. **⭐事务的概念**
   1. 事务是数据库操作序列，这些操作要么全做，要么全不做，是不可分割的工作单位。一个事务可以是一个 SQL 语句，一组 SQL 语句，或者整个程序。一般来说，一个程序包含多个事务。
   2. 事务是恢复的基本单位，也是并发控制的基本单位。
2. **定义事务的 SQL 语句**
   1. 开始语句：`BEGIN TRANSACTION;`
   2. 结束语句：`COMMIT;` // 表示提交事务的所有操作
   3. 或者 `ROLLBACK;` // 表示回滚，即事务运行中发生了某种故障，需要对已完成的操作进行撤销，回滚到事务开始的状态。
3. **⭐事务的 4 个特性（ACID）：原子性、一致性、隔离性、持续性**
   1. **原子性**：一个事务要么全做，要么全不做
   2. **一致性**：事务执行的结果必须一致。如果一个事务被迫中断，即事务中的一些操作做了，剩下的还没做，且做了的对数据库造成的修改已经存入物理数据库的话，就会造成数据库处于不一致的状态。因此事务要么全做，要么全不做，一致性和原子性是密切相关的
   3. **隔离性**：一个事务的执行不能被其他事务干扰
   4. **持续性（也称为永久性）**：一个事务一旦提交，它对数据库中数据的改变应该是永久性的
4. **⭐数据库系统的故障种类：事务内部的故障、系统故障、介质故障、计算机病毒**
   1. **事务内部的故障**
      事务内部的故障有的是可以通过事务程序本身发现的，有的是非预期的、不能由事务程序处理的。
      事务故障的情况下，数据库可能处于不正确的状态。恢复程序要在不影响其他事务运行的情况下，强行回滚该事务，即撤销该事务造成的修改，这类恢复操作称为 UNDO（事务撤销）。
   2. **系统故障（软故障）**
      系统故障是指造成系统停止运转的任何事件，使得系统要重新启动（例如硬件错误、操作系统故障、DBMS 代码错误、系统断电等等）。
      系统故障的情况下，在系统重新启动后，恢复子系统除了需要撤销未完成的事务以外，还需要 REDO（事务重做）所有已提交的事务。
   3. **介质故障（硬故障）**
      外存故障，如磁盘损坏、磁头碰撞、瞬时强磁场干扰等。
   4. **计算机病毒**
5. **⭐数据库恢复技术：数据转储、登记日志文件**
   恢复机制涉及的两个关键问题：
   1. 如何建立冗余数据
   2. 如何利用这些冗余数据实施数据库的恢复
   **恢复的基本原理**：利用存储在后备副本、日志文件和数据库镜像中的冗余数据来重建数据库。
   建立冗余数据的技术包括：
   1. **数据转储**
      数据库管理员定期将整个数据库复制到磁带、磁盘或其他存储介质上保存起来，这些备用数据称为后备副本或后援副本。当数据库遭到破坏后可以将后备副本重新装入，但重装后备副本只能将数据库恢复到存储时的状态，要想恢复到故障发生时的状态，则必须重新运行自转储后的所有更新事务。
      数据转储分为：静态转储（在系统中无运行事务时进行）、动态转储（转储期间允许对数据库进行存取或修改）。
      数据转储也可分为：海量转储（每次转储全部数据库）、增量转储（每次只转储上一次转储后更新过的数据）。
      数据转储方法分为：动态海量转储、动态增量转储、静态海量转储、静态增量转储。
   2. **登记日志文件**
      日志文件是用来记录事务对数据库的更新操作的文件，有两种格式：以记录为单位、以数据块为单位。
      - **以记录为单位的日志文件**
         需要登记的内容：各个事务的开始（`BEGIN TRANSACTION`）标记、各个事务的结束（`COMMIT`/`ROLLBACK`）标记、各个事务的所有更新操作。
         日志记录的内容：事务标识（标明哪个事务）、操作的类型（增删改）、操作对象（记录内部标识）、更新前数据的旧值（对插入而言，此项为空值）、更新后数据的新值（对删除而言，此项为空值）。
      - **以数据块为单位的日志文件**
         日志记录的内容：事务标识、被更新的数据块。
      **日志文件的作用**：
      1. 事务故障恢复和系统故障恢复必须用日志文件
      2. 在动态转储方式中必须建立日志文件，后备副本和日志文件结合才能有效恢复数据库
      3. 在静态转储方式中也可以建立日志文件，当数据库毁坏后可以重新装入后备副本把数据库恢复到转储结束时刻的正确状态，然后利用日志文件，把已完成的事务进行重做处理，对故障发生时尚未完成的事务进行撤销处理
      
      **登记日志文件的两条原则**：
      1. 登记的次序严格按并发事务执行的时间次序
      2. 必须先写日志文件，后写数据库
6. **⭐不同的数据库系统故障下的恢复策略**
   1. **事务故障的恢复**
      由系统自动完成，对用户透明：利用日志文件撤销（UNDO）此事务已对数据库进行的修改。
   2. **系统故障的恢复**
      由系统重新启动时自动完成，不需要用户干预：撤销（UNDO）故障发生时未完成的事务，重做（REDO）已完成的事务。
   3. **介质故障的恢复**
      最严重的一种故障：重装数据库，重做（REDO）已完成的事务。
      - 为避免磁盘介质出现故障影响数据库，许多数据库管理系统提供了数据库镜像功能（复制数据实现）。
7. **⭐具有检查点的恢复技术**
   这种技术在日志文件中增加了检查点记录：检查点之前的事务已经做完了，不需要重做，检查点期间和之后的事务需要重做，可以改善恢复效率。

# 第十一章：并发控制
1. **⭐并发控制**
   为了保证事务的隔离性和一致性，数据库管理系统需要对并发操作进行正确调度。
   并发控制的主要技术有：**封锁、时间戳、乐观控制法、多版本并发控制**等。
   并发操作带来的数据不一致性：
   1. **丢失修改**：两个以上的事务读入同一数据并修改，后提交的事务的提交结果破坏了先提交的事务的提交结果，导致先提交的事务对数据库的修改被丢失
   2. **不可重复读**：指事务 T1 读取数据后，事务 T2 执行更新操作，使 T1 无法再现前一次读取结果
   3. **读脏数据**：事务 T1 修改某一数据后并将其写回磁盘，事务 T2 读取同一数据后，T1 由于某种原因被撤销，这时 T1 修改过的数据恢复原值，T2 读到的数据就与数据库中的数据不一致
2. **⭐并发控制的基本方法：封锁（X 锁、S 锁）**
   1. **排他锁/写锁，X 锁**
      若事务 T 对数据对象 A 加上 X 锁，则事务 T 可以读取和修改 A。
      事务 T 释放 A 上的锁之前：其他事务不能再对 A 加任何类型的锁（保证了其他事务不能读取和修改 A）。
   2. **共享锁/读锁，S 锁**
      若事务 T 对数据对象 A 加上 S 锁，则事务 T 可以读取 A，但不能修改 A。
      事务 T 释放 A 上的 S 锁之前：其他事务可以对 A 加 S 锁，但不能对 A 加 X 锁（保证了其他事务可以读取但不能修改 A）。

4. **活锁和死锁**
   1. **活锁**
      事务 T1 封锁了数据 R，事务 T2 请求封锁 R，于是 T2 等待；T3 也请求封锁 R。T1 释放 R 上的封锁后，系统首先批准了 T3 的请求，T2 仍然等待；如果后续又不断批准 T4、T5 等事务的请求，T2 就可能永远等待，产生活锁。
      **避免活锁**：采用先来先服务（FIFO）的策略。
   2. **死锁**
      事务 T1 封锁了数据 R1，事务 T2 封锁了数据 R2；T1 又请求封锁 R2，因为 T2 已经封锁 R2，所以 T1 等待 T2 释放 R2 上的锁；接着 T2 又申请封锁 R1，因为 T1 已经封锁 R1，所以 T2 也只能等待 T1 释放 R1 上的锁。这样 T1 在等待 T2，T2 又在等待 T1，两个事务都无法结束，形成死锁。
      
      **解决死锁的方法：**
      1. **死锁的预防**
         1. **一次封锁法**：每个事务必须一次性将所有要使用的数据全部加锁，否则不能继续执行。
         2. **顺序封锁法**：预先对数据对象规定一个封锁顺序，所有事务都按这个顺序实施封锁。
      2. **死锁的诊断与解除**
         1. **超时法**：如果一个事务的等待时间超过规定时限，就认为发生了死锁。
         2. **等待图法**：构建事务等待图，节点表示正在运行的事务，边表示事务等待关系；如果等待图中出现回路，则说明发生了死锁。
         3. **解除方法**：发现死锁后，通常选择一个处理死锁代价最小的事务，将其撤销（UNDO），释放该事务持有的所有锁，使其他事务继续执行。
5. **⭐可串行化调度**
   多个事务并发执行的结果，如果与这些事务按某种次序串行执行的结果相同，则称该调度是**可串行化调度**。
   一个给定的并发调度，当且仅当它是可串行化的，才认为是正确调度。
   **举例**：现在有事务 T1 和 T2，若二者并发执行的结果与「先 T1 后 T2」相同，或者与「先 T2 后 T1」相同，则该调度是正确的；否则就是错误的。
6. **⭐冲突可串行化调度**
   **冲突操作**：不同事务对同一个数据对象的读写操作和写写操作。
   1. 读写冲突：一个事务读某数据，另一个事务写同一数据。
   2. 写写冲突：两个事务都写同一数据。
   
   不同事务的冲突操作，以及同一事务中的两个操作，都是不能交换执行顺序的；如果交换，就可能导致错误结果。
   如果一个调度在保证冲突操作顺序不变的前提下，可以通过交换非冲突操作转换成某个串行调度，则称该调度为**冲突可串行化调度**。
   冲突可串行化调度一定是可串行化调度；但可串行化调度不一定都满足冲突可串行化。
7. **⭐两段锁协议（2PL）**
   两段锁协议要求所有事务必须分两个阶段对数据项加锁和解锁：
   1. **获得封锁（扩展阶段）**：在对任何数据进行读、写操作之前，事务首先要获得该数据的封锁。
   2. **释放封锁（收缩阶段）**：在释放一个封锁之后，事务不能再申请和获得任何其他封锁。
   
   若并发执行的所有事务都遵守两段锁协议，则这些事务的任何并发调度策略都是可串行化的。
# 关系理论

1. **⭐函数依赖**
   1. **非平凡的函数依赖**：$X \to Y$，且 $Y \nsubseteq X$
   2. **平凡的函数依赖**：$X \to Y$，且 $Y \subseteq X$
      无特殊说明下，均讨论非平凡的函数依赖。即 $X$ 可以推出 $Y$，但 $Y$ 不是 $X$ 的子集；因为一般某个集合当然能推出其子集，这种情况就是平凡的函数依赖，实际意义不大。
   3. **完全函数依赖**：$X \to Y$，并且对于 $X$ 的任意真子集 $X'$，都有 $X' \nrightarrow Y$，则称 $Y$ 完全函数依赖于 $X$，记作 $X \xrightarrow{F} Y$。
   4. **部分函数依赖**：$Y$ 不完全函数依赖于 $X$，记作 $X \xrightarrow{P} Y$。例如 $A \to C$，又有 $AB \to C$，那么 $C$ 就是部分函数依赖于 $AB$ 的，这种情况会造成数据冗余。
2. **⭐码**
   1. **候选码**：是一个属性组（或者属性），通过该属性组能推出所有的属性，并且该属性组的任意真子集都不能再推出所有属性。即候选码既要能推出全体属性，又必须是最小的属性组。
   2. **求所有候选码的方法**
      1. **例**：属性集合 $U=\{A,B,C,D,E,G\}$，函数依赖集 $F=\{AB \to C,\ CD \to E,\ E \to A,\ A \to G\}$。
      2. **Step 1：找出属性类型**
         1. **一定属于候选码的属性**：只出现在函数依赖左边，或者左右两边都没出现
         2. **可能属于候选码的属性**：左右两边都出现
         3. **不属于候选码的属性**：只出现在函数依赖右边
      3. **例题分析**
         1. 只出现在左边的是 $B$ 和 $D$，没有左右两边都没出现的属性，所以 $BD$ 一定是属于候选码的属性。
         2. 左右都出现的是 $A$、$C$、$E$，因此这三个是可能属于候选码的属性，即待定备选。
         3. 只出现在右边的是 $G$，因此 $G$ 是不属于候选码的属性，可以不管。
      4. **Step 2：求闭包**
         1. 先对确定的属性求闭包，看能不能构成候选码；再将确定的属性和待定的属性进行组合，做闭包运算，直到得到闭包属性组能够推出全部属性。
         2. **闭包运算**
            1. 若要求属性组的闭包，首先设有集合 $X$，令 $X=\{该属性组\}$。
            2. $X^{(0)}$ 为自身。
            3. $X^{(1)}$ 为 $X^{(0)}$ 中的属性所能推出的属性集合。
            4. 当 $X^{(1)} \ne X^{(0)}$ 时，继续令 $X^{(2)}$ 为 $X^{(1)}$ 中的属性所能推出的属性集合。
            5. 依次类推，直到 $X^{(s)}=U$ 或者 $X^{(s)}=X^{(s-1)}$，就求得了属性组的闭包 $(X)^+$。
         3. **闭包的作用**：闭包运算可用于判断 $X \to Y$ 是否成立；当 $Y \subseteq (X)^+$ 时，有 $X \to Y$。
      5. **例题继续分析**
         1. 根据 Step 1 的分析，一定属于候选码的是 $BD$，可能属于候选码的是 $A$、$C$、$E$。
         2. 于是先对 $BD$ 求闭包，这里可求得 $BD$ 推不出全部属性，因此再分别对 $BDA$、$BDC$、$BDE$ 进行闭包运算，看其是否能得到全部属性；如若不能，再继续增加 $BDAC$、$BDAE$ 之类的组合，直到求出候选码为止。
         3. 以 $BDA$ 为例，设 $X=\{B,D,A\}$：
            1. $X^{(0)}=BDA$
            2. $X^{(1)}=BDACG$
            3. 因为 $X^{(1)} \ne X^{(0)}$，所以继续求得 $X^{(2)}=BDACGE=U$
            4. 因此 $(BDA)^+=U$，所以 $BDA$ 是候选码。
         4. 全部进行闭包运算后，可知集合 $U$ 在 $F$ 下的候选码为 $\{BDA,BDC,BDE\}$。
   3. **超码**：能推出所有属性的属性组。根据概念可知，候选码是极小的超码集，是超码的子集。
   4. **主码**：当有多个候选码时，挑出一个作为主码，简称码。
   5. **主属性**：包含在任一候选码中的属性。本例中 $A$、$B$、$C$、$D$、$E$ 都是主属性，其中 $B$、$D$ 出现在所有候选码中。
   6. **非主属性**：不包含在任何一个候选码中的属性。本例中 $G$ 是非主属性。
   7. **外码**：关系模式 $R$ 中，若有一个属性或属性组 $X$，它不是 $R$ 的码，但 $X$ 是另一个关系模式 $S$ 的码，则称 $X$ 是 $R$ 的外码。
   8. **全码**：最极端情况下，整个属性组都是码，称为全码。
3. **⭐范式**
   1. **1NF（第一范式）**
      1. 要求：所有属性都是不可再分的数据项。
      2. 如果某个属性还可以继续拆分，例如“学校”还可以拆成“高中”和“大学”，就不满足 1NF。
      3. 1NF 是关系数据库中关系模式需要满足的最低要求。
   2. **2NF（第二范式）**
      1. 要求：在满足 1NF 的前提下，不包含非主属性对码的部分函数依赖。也就是每一个非主属性都完全函数依赖于码。
      2. **例**：在关系 $R$ 中，码是“学号 + 班级”，非主属性是“姓名”。如果只通过“学号”就能推出“姓名”，不需要“班级”，则“姓名”部分函数依赖于码，不满足 2NF。
   3. **3NF（第三范式）**
      1. 要求：在满足 2NF 的前提下，不包含非主属性对码的传递函数依赖。也就是码应该直接决定非主属性，不能间接决定。
      2. **传递函数依赖**：若 $X \to Y$，$Y \to Z$，且 $Y \nrightarrow X$、$Z \nsubseteq Y$，则有 $X \to Z$，此时称 $Z$ 对 $X$ 有传递函数依赖。
      3. **例**：在关系 $R$ 中，码是“客户姓名”，非主属性是“订单编号”和“订单负责人”。通过“客户姓名”可以推出“订单编号”，再通过“订单编号”推出“订单负责人”，此时“客户姓名”和“订单负责人”是间接决定的，存在传递函数依赖，不满足 3NF。
   4. **BCNF（巴斯-科德范式）**
      1. 要求：消除任何属性对候选码的传递依赖；可以理解为每一个决定因素都包含码。
      2. 表现：在函数依赖集中，每个非平凡函数依赖 $X \to Y$ 的左边 $X$ 都必须包含候选码（整个属性组），即 $X$ 至少是一个超码。
   5. **4NF（第四范式）**
      1. 一般不作为重点，核心是：不允许存在非平凡且非函数依赖的多值依赖。
      2. **多值依赖**：设 $X,Y,Z \subseteq U$，且 $Z=U-X-Y$。当给定一组 $(x,z)$ 的值时，可以确定一组 $Y$ 的值，且这组 $Y$ 的值仅取决于 $X$，与 $Z$ 无关，则称 $Y$ 多值依赖于 $X$，记作 $X \twoheadrightarrow Y$。
         个人理解：本质上是存在一对多关系，即一个 $X$ 和一组 $Y$ 有关，但 $X$ 不能唯一确定一个 $Y$；可以通过 $X$ 找到一组 $Y$，但不一定只能确定单个值。
      3. **平凡的多值依赖**：按 $Z=U-X-Y$ 记法，若 $Z$ 是空集，则为平凡的多值依赖。
      4. **非平凡的多值依赖**：按 $Z=U-X-Y$ 记法，若 $Z$ 不是空集，则为非平凡的多值依赖。
      5. 简单记：同一个 $X$ 可以独立对应多个 $Y$ 和多个 $Z$，组合时容易产生冗余。
   6. **范式判断方法与分解方法**
      1. **判断 2NF**：先找码，再看是否存在非主属性对码的部分函数依赖。
         1. 例：$R=\{A,B,C,D\}$，若码是 $AB$，函数依赖集中有 $A \to C$、$AB \to D$，则对于 $C$，只需要 $A$ 就能推出，不需要完整的 $AB$，所以 $C$ 部分函数依赖于 $AB$，这种情况不满足 2NF。
         2. 若要分解为 2NF，只需将不符合要求的部分拆出来，即分为 $R_1(A,B,D)$ 和 $R_2(A,C)$。
      2. **判断 3NF**：在没有部分函数依赖的基础上，再看是否存在非主属性对码的传递函数依赖。
         1. 例：若码是 $AB$，函数依赖集 $F=\{AB \to C,\ C \to D\}$。这里不存在部分函数依赖，但对于 $D$，需要 $AB$ 推出 $C$ 后，才能通过 $C$ 间接推出 $D$，所以 $D$ 传递函数依赖于 $AB$，不满足 3NF。
         2. 若要分解为 3NF，同样将不符合要求的部分拆出来，即分为 $R_1(A,B,C)$ 和 $R_2(C,D)$。
      3. **判断 BCNF**：在没有部分函数依赖的基础上，还要求每一个决定因素都包含码，即每个非平凡函数依赖左边都至少是超码。
         1. 例：$R=\{A,B,C\}$，$F=\{AC \to B,\ AB \to C,\ B \to C\}$，候选码为 $AC$ 和 $AB$。
         2. 这里不存在部分函数依赖，但是对于 $B \to C$ 来说，$B$ 没有包含码，也不是超码，因此不满足 BCNF。
4. **⭐最小函数依赖集**
   1. **求最小函数依赖集的方法**
      1. **Step 1：拆分右侧**
         1. 将函数依赖右侧拆成单属性。
         2. 例如将 $A \to BC$ 拆为 $A \to B$ 和 $A \to C$。
      2. **Step 2：去除自身求闭包**
         1. 若有 $AB \to C$、$BC \to E$、$AE \to G$，判断 $AB \to C$ 是否冗余时，先删除 $AB \to C$，再基于剩余依赖求 $AB$ 的闭包。
         2. 若 $AB$ 通过剩余依赖仍能推出 $C$，则删除 $AB \to C$ 这个依赖关系。
      3. **Step 3：左侧最小化**
         1. 对左侧含有多个属性的函数依赖，逐个尝试删除左侧属性。
         2. 例如当前保留关系有 $ABC \to D$，观察左侧 $ABC$ 中：
            1. $A$ 是否能由 $BC$ 推出
            2. $B$ 是否能由 $AC$ 推出
            3. $C$ 是否能由 $AB$ 推出
         3. 假设 $C$ 能被 $AB$ 推出，那么左侧可以去掉 $C$，将 $ABC \to D$ 更新为 $AB \to D$。
      4. **例题**：设 $F=\{C \to A,\ CG \to BD,\ CE \to A,\ ACD \to B\}$，求最小函数依赖集。
         1. **Step 1：拆分右侧**
            1. 将 $CG \to BD$ 拆分为 $CG \to B$ 和 $CG \to D$。
            2. 拆分后：$F=\{C \to A,\ CG \to B,\ CG \to D,\ CE \to A,\ ACD \to B\}$。
         2. **Step 2：去除自身求闭包，删除冗余依赖**
            1. 去掉 $C \to A$ 后，$C^+=C$，不能推出 $A$，因此保留 $C \to A$。
            2. 去掉 $CG \to B$ 后，$(CG)^+=CGADB$，仍能推出 $B$，因此删除 $CG \to B$。
            3. 去掉 $CG \to D$ 后，$(CG)^+=CGAB$，不能推出 $D$，因此保留 $CG \to D$。
            4. 去掉 $CE \to A$ 后，$(CE)^+=CEA$，仍能推出 $A$，因此删除 $CE \to A$。
            5. 去掉 $ACD \to B$ 后，$(ACD)^+=ACD$，不能推出 $B$，因此保留 $ACD \to B$。
         3. **Step 3：左侧最小化**
            1. $C \to A$ 已经是最小左侧。
            2. $CG \to D$ 已经是最小左侧。
            3. 对 $ACD \to B$，因为 $C \to A$，所以 $C$ 可以推出 $A$，左侧中的 $A$ 冗余，去掉 $A$，更新为 $CD \to B$。
         4. **结果**：本题的最小函数依赖集为 $\{C \to A,\ CG \to D,\ CD \to B\}$。
5. **模式分解**
   1. **模式分解的准则**：无损连接、保持函数依赖。
   2. **无损连接**：分解后再次自然连接，结果与分解前相同。
   3. **判断无损连接的方法（表格法）**
      1. **Step 1：画表格**
         1. 列表示所有属性，有多少个属性就画多少个属性列。
         2. 行表示分解后的关系，有几个关系就画几行。
      2. **Step 2：填初始标记**
         1. 根据每一行关系进行判断，关系中包含的属性在对应列标为 $a_j$，其中 $j$ 表示该属性所在列号。
         2. 关系中不包含的属性标为 $b_{ij}$，其中 $i$ 表示行号，$j$ 表示列号。
      3. **Step 3：根据函数依赖更新表格**
         1. 依次考察函数依赖集里的各个依赖关系，例如 $X \to Z$。
         2. 若若干行在 $X$ 对应属性列上的标记相同，则查看这些行在 $Z$ 对应属性列上的标记。
         3. 如果这些标记中有 $a_j$，则将对应 $Z$ 列的标记全部改为 $a_j$。
         4. 如果没有 $a_j$，则找下标最小的 $b_{ij}$，将这些标记全部改为该 $b_{ij}$。
      4. **Step 4：反复执行上述操作**
         1. 重复 Step 3，直到表格不再变化。
         2. 若出现某一行全部变为 $a$，则表明该分解具有无损连接性；否则不具有无损连接性。
   4. **例题**
      1. 函数依赖集：$F=\{A \to C,\ C \to D,\ B \to C,\ DE \to C,\ CE \to A\}$。
      2. 分解为：$R_1(AD)$、$R_2(AB)$、$R_3(BC)$、$R_4(CDE)$、$R_5(AE)$。

         | $R_5$ | $a_1$ | $b_{52}$ | $b_{53}$ | $b_{54}$ | $a_5$ |

      4. **更新表格过程**
         1. 根据 $A \to C$：$R_1$、$R_2$、$R_5$ 的 $A$ 列同为 $a_1$，因此统一它们的 $C$ 列；该组中没有 $a_3$，所以统一为下标最小的 $b_{13}$。

         2. 根据 $C \to D$：$C$ 列相同的行需要统一 $D$ 列；对应组中含有 $a_4$，所以相关 $D$ 列统一为 $a_4$。

      1. 已知属性集 $U$、函数依赖集 $F$，模式分解后应该得到多个 $U$ 的子集。
      2. **Step 1：求最小函数依赖集**
         1. 求出 $F_{min}$。
      3. **Step 2：找未出现属性**
         1. 观察 $U$，找出其中未在 $F_{min}$ 中出现过的属性，将其单独分为一个集合。
         2. 例如 $U$ 中有 $A,B,C,D,E,F,G$，但 $F_{min}$ 中没有关于 $G$ 的依赖关系，则划分出 $\{G\}$。
      4. **Step 3：按相同决定因素分组**
         1. 观察 $F_{min}$，若有多个依赖关系的决定因素（即左侧）相同，则都划分到同一个集合中。
         2. 若没有相同的，则仅将该依赖关系涉及的属性划分到同一个集合。
         3. 例如 $F_{min}$ 中有 $A \to B$、$A \to C$、$D \to E$，则划分出 $\{A,B,C\}$ 和 $\{D,E\}$。
      5. **Step 4：补候选码**
         1. 求出 $F_{min}$ 的候选码，若候选码未在上述分类中出现，则单独将候选码分为一类。
         2. 例如上述 $F_{min}$ 的候选码为 $ADG$，它未完整出现在各分类中，因此再划分出集合 $\{A,D,G\}$。
      6. **最终结果**
         1. 由上述举例可知，最终的模式分解为 $\{A,B,C\}$、$\{D,E\}$、$\{A,D,G\}$。

# 关系语言
1. **⭐关系代数语言**
   1. **集合运算符**（设有关系 $R$ 和关系 $S$）

      | 运算符 | 写法 | 含义 |
      | --- | --- | --- |
      | 并 $\cup$ | $R \cup S$ | 由属于 $R$ 或属于 $S$ 的元组构成，同时去掉重复元组 |
      | 差 $-$ | $R-S$ | 由属于 $R$ 但不属于 $S$ 的元组构成 |
      | 交 $\cap$ | $R \cap S$ | 由既属于 $R$ 又属于 $S$ 的元组构成 |
      | 笛卡尔积 $\times$ | $R \times S$ | 由 $R$ 中的每个元组与 $S$ 中的所有元组进行组合 |
   2. **关系运算符**

      | 运算符 | 写法 | 含义 |
      | --- | --- | --- |
      | 选择 $\sigma$ | $\sigma_{条件}(表名)$ | 得到表中的指定行 |
      | 投影 $\pi$ | $\pi_{列名}(表名)$ | 得到表中的指定列，投影后要去除重复行 |
      | 连接 $\Join$ | $R \Join_{条件} S$ | 将两个表根据指定条件连接在一起 |
      | 除 $\div$ | $R \div S$ | 求满足 $S$ 中全部组合要求的 $R$ 中剩余属性 |

      **连接类型：**

      | 类型 | 写法/条件 | 含义 |
      | --- | --- | --- |
      | 等值连接 | $R.A=S.B$ | 连接条件为属性相等 |
      | 自然连接 | $R \Join S$ | 同名属性相等，并且要去掉重复列 |
      | 悬浮元组 | - | 自然连接时，由于 $S$ 中不匹配而在 $R$ 中被舍弃的元组 |
      | 外连接 | $R \mathbin{⟗} S$ | 保留悬浮元组，不匹配的位置填 `NULL` |
      | 左外连接 | $R \mathbin{⟕} S$ | 只保留 $R$ 中悬浮元组 |
      | 右外连接 | $R \mathbin{⟖} S$ | 只保留 $S$ 中悬浮元组 |
      4. **除运算 $\div$**
         1. 设 $R \div S$ 的结果为 $T$，则 $T$ 包含所有在 $R$ 中但不在 $S$ 中的属性和值。
         2. 并且 $T$ 的元组与 $S$ 的元组经过组合后，均能出现在 $R$ 中。
         3. **例**：设 $R(A,B,C)$，图中 $S$ 关注可与 $R$ 匹配的属性组合 $(B,C)$，则 $R \div S$ 的结果属性为 $A$。

            | $R$ | A | B | C |
            | --- | --- | --- | --- |
            |  | $a_1$ | $b_1$ | $c_2$ |
            |  | $a_2$ | $b_3$ | $c_7$ |
            |  | $a_3$ | $b_4$ | $c_6$ |
            |  | $a_1$ | $b_2$ | $c_3$ |
            |  | $a_4$ | $b_1$ | $c_6$ |
            |  | $a_2$ | $b_2$ | $c_3$ |
            |  | $a_1$ | $b_2$ | $c_1$ |

            | $S$ | B | C | D |
            | --- | --- | --- | --- |
            |  | $b_1$ | $c_2$ | $d_1$ |
            |  | $b_2$ | $c_1$ | $d_1$ |
            |  | $b_2$ | $c_3$ | $d_2$ |

            | $T=R \div S$ | A |
            | --- | --- |
            |  | $a_1$ |

         4. 对 $a_1$ 来说，$S$ 中需要匹配的 $(B,C)$ 组合 $(b_1,c_2)$、$(b_2,c_1)$、$(b_2,c_3)$ 都能在 $R$ 中与 $a_1$ 组合出现，所以 $a_1$ 属于结果 $T$。
         5. 对 $a_2$ 来说，虽然 $a_2$ 也出现了 $S$ 中的 $(b_2,c_3)$，但 $a_2$ 与 $S$ 中其余 $(b_1,c_2)$、$(b_2,c_1)$ 的组合没有出现在 $R$ 中，所以 $a_2$ 不属于结果 $T$。
2. **⭐关系代数解题方法**
   1. **常规题（求某几个属性特定值）**
      1. 格式一般为：
         $$
         \pi_{属性列}(\sigma_{条件}(表名 \Join 表名))
         $$
      2. **投影运算**的下标是题目要求的最终需要的列。
      3. **选择运算**的下标是题目给出的属性条件。
      4. 并行条件之间用逗号隔开；条件表达式中的运算符常见有 $\land$、$\lor$、$<>$、$\ge$、$\le$、$>$、$<$、$=$。
      5. 若涉及多个表，表之间常用自然连接。
   2. **除运算题（求满足某属性全部值的其他属性）**
      1. 这类题常见关键词是“全部”，本质是求满足 $B$ 表某属性全部值的 $A$ 表上的其他属性，是除运算的典型特征。
      2. 一般做法：先分别对相关表做投影，再对生成的关系做除运算。
      3. 若 $A$ 中包含属性 $x$ 和 $y$，$B$ 中包含属性 $y$，且 $B$ 中 $y$ 的值为全集且无重复，则求全部 $x$ 可写作：
         $$
         \pi_{x,y}(A) \div \pi_y(B)
         $$
      4. **例**：$A$ 表为学生选课表，属性包括学号和所选课程；$B$ 表为课程信息表，属性包括课程。若要求“选了全部课程的学生学号”，通常写作：
         $$
         \pi_{学号,课程}(A) \div \pi_{课程}(B)
         $$
      5. **例题展开**：若 $A$ 表为学生选课表，属性包括学号和所选课程；$B$ 表为课程信息表，属性包括课程，求选了全部课程的学生学号。
         1. 全部课程只在 $B$ 表出现，学号只在 $A$ 表出现。
         2. 先取 $A$ 表中的“学号、课程”，再取 $B$ 表中的“课程”，最后二者相除：
            $$
            \pi_{学号,课程}(A) \div \pi_{课程}(B)
            $$
   3. **差运算题**
      1. **例**：有学生表 $SC$，包含属性姓名、成绩，求没有任何一门课程低于 80 分的学生姓名。
      2. 思路：先求出有课程低于 80 分的学生姓名，再用全体学生姓名相减。
      3. 关系代数表达式：
         $$
         \pi_{姓名}(SC)-\pi_{姓名}(\sigma_{成绩<80}(SC))
         $$
3. **元组关系演算语言**
   1. **元组演算表达式**：$\{t\mid \varphi(t)\}$，其中 $t$ 是元组变量，$\varphi(t)$ 是公式，它由原子公式和运算符组成。
   2. **原子公式**

      | 原子公式 | 含义 |
      | --- | --- |
      | $R(t)$ | 表示 $t$ 是关系 $R$ 中的元组 |
      | $t[i]\ \theta\ u[j]$ | 表示元组 $t$ 的第 $i$ 个分量和元组 $u$ 的第 $j$ 个分量满足比较关系 $\theta$ |
      | $t[i]\ \theta\ c$ 或 $c\ \theta\ t[i]$ | 表示元组 $t$ 的第 $i$ 个分量和常量 $c$ 满足比较关系 $\theta$ |
   3. **运算符**（优先级从高到低）

      | 类型 | 运算符 | 优先级说明 |
      | --- | --- | --- |
      | 比较运算符 | $>$、$=$、$<$ 等 | 优先级最高 |
      | 量词运算符 | $\exists$、$\forall$ | $\exists$ 的优先级大于 $\forall$ |
      | 逻辑运算符 | $\neg$、$\land$、$\lor$ | $\neg$ > $\land$ > $\lor$ |
4. **关系代数语言和元组演算语言的转换**
   1. **并**
      $$
      R\cup S=\{t\mid R(t)\lor S(t)\}
      $$
   2. **交**
      $$
      R\cap S=\{t\mid R(t)\land S(t)\}
      $$
   3. **差**
      $$
      R-S=\{t\mid R(t)\land \neg S(t)\}
      $$
   4. **笛卡尔积**
      $$
      R\times S=\{t^{(n+m)}\mid (\exists u^{(n)})(\exists v^{(m)})(R(u)\land S(v)\land t[1]=u[1]\land\cdots\land t[n]=u[n]\land t[n+1]=v[1]\land\cdots\land t[n+m]=v[m])\}
      $$
      其中 $R$ 有 $n$ 个属性，$S$ 有 $m$ 个属性。根据笛卡尔积定义，结果元组 $t$ 的目数为 $n+m$，即共有 $n+m$ 个属性。
   5. **投影**
      $$
      \pi_{i_1,i_2,\cdots,i_k}(R)=\{t^{(k)}\mid (\exists u)(R(u)\land t[1]=u[i_1]\land\cdots\land t[k]=u[i_k])\}
      $$
      表示最终需要 $k$ 列，所以 $t$ 的目数为 $k$。令中间变量 $u$ 表示 $R$ 中的全部元组，结果元组 $t$ 的第 1 列对应 $R$ 中需要的第 $i_1$ 列，最后一列对应 $R$ 中的第 $i_k$ 列。
   6. **选择**
      $$
      \sigma_F(R)=\{t\mid R(t)\land F'\}
      $$
      $F$ 是选择条件，$F'$ 是将 $F$ 等价替换后的元组演算表达式。
5. **元组关系演算解题例题**
   1. **查询 Student 表中 IS 系的全体学生**
      1. 已知学生所在系为第 5 列属性，表达式为：
         $$
         S_{IS}=\{t\mid Student(t)\land t[5]='IS'\}
         $$
      2. 上式表示结果集为 $S_{IS}$，其中的元组 $t$ 满足两个条件：$t$ 属于 `Student` 表，并且第 5 列的值为 `IS`。
   2. **查询 Student 表中学生的姓名和所在系**
      1. 已知姓名为第 2 列，所在系为第 5 列，表达式为：
         $$
         S_1=\{t^{(2)}\mid (\exists u)(Student(u)\land t[1]=u[2]\land t[2]=u[5])\}
         $$
      2. 上式表示结果集为 $S_1$，其中每个元组 $t$ 有两列属性：
         1. 结果集第 1 列 $t[1]$ 对应 `Student` 表中的第 2 列 $u[2]$，即姓名。
         2. 结果集第 2 列 $t[2]$ 对应 `Student` 表中的第 5 列 $u[5]$，即所在系。
6. **元组关系演算解题格式**
   1. 首先设结果集，例如设为 $S$，令其中的元组为 $t$。
   2. 若题目中指明了需要哪些属性，需要标注 $t$ 的目数。
   3. 当需要使用量词运算符时，记得前后用括号括起来。
   4. 各条件之间一般用交运算，即逻辑与 $\land$ 连接。
   5. 在元组表达式中，记得首先指出所设元组属于哪个关系。
   6. 通用格式：
      $$
      S=\{t\mid (量词运算)(指出元组所属的关系\land 元组需要满足的条件)\}
      $$
   7. **补充**：把不产生无限关系的表达式称为**安全表达式**，所采取的措施称为**安全措施**。
# SQL 语言
1. **⭐CREATE/DROP/ALTER 语句**
   1. **定义模式：CREATE SCHEMA**
      1. 格式：
         ```sql
         CREATE SCHEMA 模式名 AUTHORIZATION 用户名;
         ```
      2. 例：为用户 `WANG` 定义一个 `S-T` 模式：
         ```sql
         CREATE SCHEMA "S-T" AUTHORIZATION WANG;
         ```
      3. 若未指定模式名，则默认模式名为用户名：
         ```sql
         CREATE SCHEMA AUTHORIZATION WANG;
         ```
   2. **删除模式：DROP SCHEMA**
      1. 级联删除：删除该模式，同时把该模式中的所有表、视图等对象一并删除。
         ```sql
         DROP SCHEMA 模式名 CASCADE;
         ```
      2. 限制删除：若该模式下已经定义了表或视图等对象，则拒绝执行删除语句。
         ```sql
         DROP SCHEMA 模式名 RESTRICT;
         ```
   3. **定义基本表：CREATE TABLE**
      1. 格式：
         ```sql
         CREATE TABLE 表名
         (
            列名1 数据类型 [列级完整性约束条件],
            列名n 数据类型 [列级完整性约束条件],
            表级完整性约束条件1,
            表级完整性约束条件n
         );
         ```
      2. 注意：如果没有列级完整性的约束条件，可以不写。
      3. **常见数据类型**

         | 数据类型 | 含义 |
         | --- | --- |
         | `CHAR(n)` | 长度为 `n` 的字符型 |
         | `VARCHAR(n)` | 最大长度为 `n` 的变长字符型 |
         | `NUMBER(n)` | 长度为 `n` 的数字型 |
         | `INT` | 长整型，通常占 4B |
         | `SMALLINT` | 短整型，通常占 2B |
         | `BIGINT` | 大整型，通常占 8B |
         | `FLOAT(n)` | 精度至少为 `n` 位数字的浮点数 |
         | `DATE` | 日期，格式为 `YYYY-MM-DD` |
         | `TIME` | 时间，格式为 `HH:MM:SS` |
      4. **列级完整性约束条件**

         | 约束 | 含义 |
         | --- | --- |
         | `PRIMARY KEY` | 主码；当只有一个主码时，可直接在对应的属性列标注 |
         | `NOT NULL` | 非空，表示该属性列不能取空值 |
         | `UNIQUE` | 唯一值，表示该属性列只能取唯一值 |
         | `CHECK(条件)` | 检查该列是否满足某个条件，例如 `CHECK(成绩 >= 0)` |

      5. **表级完整性约束条件**

         | 约束 | 含义 |
         | --- | --- |
         | `PRIMARY KEY(列名1, 列名n)` | 实体完整性；当主码由多个属性构成时，必须作为表级完整性进行定义 |
         | `FOREIGN KEY(列名1) REFERENCES 被参照表(列名1)` | 参照完整性；用于定义外码及其参照的表和属性 |

      6. **CREATE TABLE 示例**
         ```sql
         CREATE TABLE TAB1
         (
            Sno VARCHAR(10),
            Cno NUMBER(10),
            Grade INT NOT NULL,
            PRIMARY KEY(Sno, Cno),
            FOREIGN KEY(Sno) REFERENCES TAB2(Sno)
         );
         ```
         1. `PRIMARY KEY(Sno, Cno)`：表示 `Sno` 和 `Cno` 共同组成复合主码，所以必须写成表级完整性约束。
         2. `FOREIGN KEY(Sno) REFERENCES TAB2(Sno)`：表示 `TAB1` 中的 `Sno` 参照 `TAB2` 中的 `Sno`。
   4. **在模式中定义表**
      一个模式可以包含多种基本表，常见有三种方式在模式中定义基本表。

      | 方式 | SQL 格式 | 说明 |
      | --- | --- | --- |
      | 创建表时指出模式 | `CREATE TABLE 模式名.表名 (...);` | 直接在表名前写出所属模式 |
      | 创建模式时直接定义表 | `CREATE SCHEMA 模式名 AUTHORIZATION 用户名 CREATE TABLE 表名 (...);` | 创建模式的同时创建基本表 |
      | 设置所属模式 | 创建表时不写模式名 | 先设置当前所属模式，之后创建表时可省略模式名 |

      示例：
      ```sql
      CREATE TABLE 模式名.表名
      (
         列定义语句,
         完整性约束语句
      );
      ```
      ```sql
      CREATE SCHEMA 模式名 AUTHORIZATION 用户名
      CREATE TABLE 表名
      (
         列定义语句,
         完整性约束语句
      );
      ```
   5. **修改基本表：ALTER TABLE**

      | 操作 | SQL 格式 | 说明/例子 |
      | --- | --- | --- |
      | 增加新的属性列 | `ALTER TABLE 表名 ADD 新列名 数据类型 完整性约束条件;` | 例：`ALTER TABLE SC ADD COLUMN Time DATE;` |
      | 增加列级完整性约束条件 | `ALTER TABLE 表名 ADD 列级完整性约束条件;` | 例：`ALTER TABLE SC ADD UNIQUE(Cname);` |
      | 增加表级完整性约束条件 | `ALTER TABLE 表名 ADD 表级完整性约束条件;` | 例：`ALTER TABLE SC ADD FOREIGN KEY(Cno) REFERENCES Student(Cno);` |
      | 删除列（级联） | `ALTER TABLE 表名 DROP 列名 CASCADE;` | 引用了该列的其他对象（例如视图）一起删除 |
      | 删除列（限制） | `ALTER TABLE 表名 DROP 列名 RESTRICT;` | 若该列被其他对象引用，则拒绝删除 |
      | 删除完整性约束（级联） | `ALTER TABLE 表名 DROP CONSTRAINT 完整性约束名 CASCADE;` | 级联删除相关依赖 |
      | 删除完整性约束（限制） | `ALTER TABLE 表名 DROP CONSTRAINT 完整性约束名 RESTRICT;` | 若存在依赖则拒绝删除 |
      | 修改列 | `ALTER TABLE 表名 ALTER COLUMN 列名 数据类型;` | 例：`ALTER TABLE SC ALTER COLUMN Sage INT;` |

      例：向 `SC` 表中增加时间列，数据类型为日期型：
      ```sql
      ALTER TABLE SC ADD COLUMN Time DATE;
      ```
      例：将 `SC` 表中原有的 `Sage` 修改为整型：
      ```sql
      ALTER TABLE SC ALTER COLUMN Sage INT;
      ```
   6. **删除基本表：DROP TABLE**

      | 操作 | SQL 格式 | 说明 |
      | --- | --- | --- |
      | 级联删除 | `DROP TABLE 表名 CASCADE;` | 删除该表时，相关依赖对象（例如视图）也会被删除 |
      | 限制删除 | `DROP TABLE 表名 RESTRICT;` | 若被其他表约束引用，或存在相关视图等对象，则不能删除 |

   7. **索引**
      1. **建立索引：CREATE UNIQUE/CLUSTER INDEX**

         | 类型 | SQL 格式 | 说明/例子 |
         | --- | --- | --- |
         | 唯一索引 | `CREATE UNIQUE INDEX 索引名 ON 表名(列名1 次序, 列名n 次序);` | 例：`CREATE UNIQUE INDEX SCno ON SC(Sno ASC, Cno DESC);` |
         | 聚簇索引 | `CREATE CLUSTER INDEX 索引名 ON 表名(列名1 次序, 列名n 次序);` | 建立聚簇索引 |

      2. **修改索引（重命名）：ALTER INDEX**
         ```sql
         ALTER INDEX 旧索引名 RENAME TO 新索引名;
         ```
         例：将 `SC` 表的 `SCno` 索引名改为 `SCSno`：
         ```sql
         ALTER INDEX SCno RENAME TO SCSno;
         ```
      3. **删除索引：DROP INDEX**
         ```sql
         DROP INDEX 索引名;
         ```
         例：删除 `SC` 表的 `SCSno` 索引：
         ```sql
         DROP INDEX SCSno;
         ```
2. **⭐SELECT 语句**
   1. **一般格式**
      ```sql
      SELECT DISTINCT/ALL 目标列表达式
      FROM 表名/视图名
      WHERE 条件表达式
      GROUP BY 列名 HAVING 条件表达式
      ORDER BY 列名 次序;
      ```

      | 子句 | 作用 |
      | --- | --- |
      | `SELECT DISTINCT/ALL 目标列表达式` | 要显示的属性列；`DISTINCT` 去重，`ALL` 保留全部结果 |
      | `FROM 表名/视图名` | 查询的对象 |
      | `WHERE 条件表达式` | 查询条件 |
      | `GROUP BY 列名 HAVING 条件表达式` | 对查询结果分组，并筛选分组 |
      | `ORDER BY 列名 次序` | 对最终查询结果排序 |

   2. **基本查询：SELECT 目标列表达式**

      | 查询目标 | SQL 格式 | 例子 |
      | --- | --- | --- |
      | 查询指定列 | `SELECT 列名1, 列名n FROM 表名;` | `SELECT X, Y FROM TAB;` |
      | 查询全部列 | `SELECT * FROM 表名;` | `SELECT * FROM TAB;` |
      | 查询计算后的值 | `SELECT 表达式 FROM 表名;` | `SELECT 2022-age FROM TAB;` |
      | 改变查询结果的列标题 | `SELECT 列名 别名 FROM 表名;` | `SELECT X x1, Y y1 FROM TAB;` |
      | 取消重复行 | `SELECT DISTINCT 列名 FROM 表名;` | `SELECT DISTINCT X FROM TAB;` |

      1. **查询计算后的值**：表达式可以是算术表达式、字符串常量、函数等。
         ```sql
         SELECT 2022-age
         FROM TAB;
         ```
      2. **改变查询结果的列标题**：查询 `TAB` 表中的 `X` 和 `Y` 属性列，并在结果中用别名 `x1` 和 `y1` 显示。
         ```sql
         SELECT X x1, Y y1
         FROM TAB;
         ```
      3. **取消查询结果中的重复行**：使用 `DISTINCT`；如果没有写 `DISTINCT`，默认是 `ALL`。
         ```sql
         SELECT DISTINCT X
         FROM TAB;
         ```
   3. **聚集函数**
      1. **注意点**
         1. 当聚集函数遇到空值时，除 `COUNT(*)` 外，都跳过空值，只处理非空值。
         2. 聚集函数只能用于 `SELECT` 语句和 `GROUP BY` 中的 `HAVING` 子句中。

      | 聚集函数 | 作用 | 说明 |
      | --- | --- | --- |
      | `COUNT(*)` | 统计元组个数 | 某个元组的一个或部分列为空时，不影响统计结果 |
      | `COUNT(DISTINCT/ALL 列名)` | 统计某一列值的个数 | `DISTINCT` 表示计算前取消重复值；不指定时默认为 `ALL` |
      | `AVG(DISTINCT/ALL 列名)` | 计算某一列值的平均数 | 该列必须为数值型 |
      | `SUM(DISTINCT/ALL 列名)` | 计算某一列值的总和 | 该列必须为数值型 |
      | `MAX(DISTINCT/ALL 列名)` | 计算某一列值的最大值 | 可用于数值、字符、日期等可比较类型 |
      | `MIN(DISTINCT/ALL 列名)` | 计算某一列值的最小值 | 可用于数值、字符、日期等可比较类型 |

      2. **统计元组个数**：查询表中学生的总数。
         ```sql
         SELECT COUNT(*)
         FROM SC;
         ```
      3. **计算平均值**：计算 `SC` 表中的平均成绩 `Grade`。
         ```sql
         SELECT AVG(Grade)
         FROM SC;
         ```
   4. **WHERE 条件表达式**
      1. **比较大小**

         | 比较运算符 | 含义 |
         | --- | --- |
         | `=` | 等于 |
         | `>` | 大于 |
         | `<` | 小于 |
         | `>=` | 大于等于 |
         | `<=` | 小于等于 |
         | `!=` 或 `<>` | 不等于 |
         | `!>` | 不大于 |
         | `!<` | 不小于 |

         例：查询 `SC` 表中全体计算机学生的姓名。
         ```sql
         SELECT Sname
         FROM SC
         WHERE Sdept = 'cs';
         ```

         例：查询 `TAB` 表中 `X > 20` 的 `Y`。
         ```sql
         SELECT Y
         FROM TAB
         WHERE X > 20;
         ```

      2. **确定范围**

         | 条件 | SQL 格式 | 含义 |
         | --- | --- | --- |
         | 在范围内 | `WHERE 列名 BETWEEN 最小值 AND 最大值;` | 查询列值在最小值和最大值之间的记录 |
         | 不在范围内 | `WHERE 列名 NOT BETWEEN 最小值 AND 最大值;` | 查询列值不在最小值和最大值之间的记录 |

         例：查询 `TAB` 表上 `age` 在 20 到 30 之间的人的 `name` 和 `sex`。
         ```sql
         SELECT name, sex
         FROM TAB
         WHERE age BETWEEN 20 AND 30;
         ```

         例：查询 `TAB` 表上 `age` 不在 20 到 30 之间的人的 `name` 和 `sex`。
         ```sql
         SELECT name, sex
         FROM TAB
         WHERE age NOT BETWEEN 20 AND 30;
         ```

      3. **确定集合**

         | 条件 | SQL 格式 | 含义 |
         | --- | --- | --- |
         | 属于集合 | `WHERE 列名 IN ('值1', '值n');` | 查询列值属于给定集合的记录 |
         | 不属于集合 | `WHERE 列名 NOT IN ('值1', '值n');` | 查询列值不属于给定集合的记录 |

         例：在 `SC` 表的 `Sdept` 列中，查找属于计算机专业（`CS`）、数学专业（`MA`）、信息专业（`IS`）的学生姓名 `S_name`。
         ```sql
         SELECT S_name
         FROM SC
         WHERE Sdept IN ('CS', 'MA', 'IS');
         ```

         例：在 `SC` 表的 `Sdept` 列中，查找既不是计算机专业（`CS`）也不是数学专业（`MA`）的学生姓名 `S_name`。
         ```sql
         SELECT S_name
         FROM SC
         WHERE Sdept NOT IN ('CS', 'MA');
         ```

      4. **字符匹配**
         通配符写在字符串当中，用来匹配满足特殊条件的字符串。

         | 通配符 | 含义 | 例子 |
         | --- | --- | --- |
         | `%` | 表示任意长度的字符串，长度可以为 0 | `a%b` 表示以 `a` 开头、以 `b` 结尾的任意长度字符串 |
         | `_` | 表示单个字符 | `a_b` 表示以 `a` 开头、以 `b` 结尾，且长度为 3 的字符串 |

         | 条件 | SQL 格式 | 含义 |
         | --- | --- | --- |
         | 匹配字符串 | `WHERE 列名 LIKE '字符串' ESCAPE '转义字符';` | 查询满足指定匹配串的记录 |
         | 不匹配字符串 | `WHERE 列名 NOT LIKE '字符串' ESCAPE '转义字符';` | 查询不满足指定匹配串的记录 |

         注意：在 ASCII 码表中，一个汉字的长度为 2。

         **转义字符**：字符串中紧跟在转义字符后的 `%` 或 `_`，不再具有通配符含义。设置转义字符的语句为 `ESCAPE '符号'`，该符号可以自己设置，一般采用 `\`。

         例：在 `Student` 表中查找所有姓刘且全名为 3 个汉字的学生的 `Sname`、`Sno` 和 `Ssex`。
         ```sql
         SELECT Sname, Sno, Ssex
         FROM Student
         WHERE Sname LIKE '刘__';
         ```

         例：在 `SC` 表中查找课程名 `Cname` 为 `DB_Design` 的课程号 `Cno`。
         ```sql
         SELECT Cno
         FROM SC
         WHERE Cname LIKE 'DB\_Design' ESCAPE '\';
         ```

         例：在 `Student` 表中查找所有不姓刘的学生的 `Sname`。
         ```sql
         SELECT Sname
         FROM Student
         WHERE Sname NOT LIKE '刘%';
         ```

      5. **查询空值**

         | 条件 | SQL 格式 | 含义 |
         | --- | --- | --- |
         | 查询空值 | `WHERE 列名 IS NULL;` | 查询该列为空值的记录 |
         | 查询非空值 | `WHERE 列名 IS NOT NULL;` | 查询该列不为空值的记录 |

         例：在 `SC` 表中查询缺少成绩 `Grade` 的学生姓名 `Sname`。
         ```sql
         SELECT Sname
         FROM SC
         WHERE Grade IS NULL;
         ```

         例：查询全部有成绩 `Grade` 的学生姓名 `Sname`。
         ```sql
         SELECT Sname
         FROM SC
         WHERE Grade IS NOT NULL;
         ```

      6. **多重条件查询**

         | 条件 | SQL 格式 | 说明 |
         | --- | --- | --- |
         | 与 | `WHERE 条件表达式 AND 条件表达式;` | 两个条件都满足 |
         | 或 | `WHERE 条件表达式 OR 条件表达式;` | 满足任一条件即可 |

         可以用 `AND` 或者 `OR` 将多个条件表达式组合在一起，其中 `AND` 的优先级大于 `OR`。

   5. **GROUP BY 子句**
      1. 格式：
         ```sql
         GROUP BY 列名 HAVING 条件表达式
         ```
      2. `GROUP BY` 用于将查询结果按某一列或多列的值分组，值相等的为一组。
      3. 目的是细化聚集函数的作用对象；分组后聚集函数将作用于每一个组，每一组都有一个函数值。
      4. **GROUP BY 列名**：求 `SC` 表中各课程号 `Cno` 下相应的选课人数 `Sno`。
         ```sql
         SELECT Cno, COUNT(Sno)
         FROM SC
         GROUP BY Cno;
         ```
         表示具有相同 `Cno` 值的元组为一组，对每一组用 `COUNT` 计算，求得该组人数。
      5. **GROUP BY 列名 HAVING 筛选条件**

         | 子句 | 作用 | 注意点 |
         | --- | --- | --- |
         | `HAVING` | 从分组中选择满足条件的组 | 常与聚集函数一起使用 |
         | `WHERE` | 从基本表或视图中选择满足条件的元组 | `WHERE` 子句不可以直接接聚集函数 |

         例：在 `SC` 表中查询平均成绩 `Grade` 大于等于 90 的学生学号 `Sno` 和平均成绩。
         ```sql
         SELECT Sno, AVG(Grade) AS AvgGrade
         FROM SC
         GROUP BY Sno
         HAVING AVG(Grade) >= 90;
         ```
         注意：这里查询的是“平均成绩”，所以 `SELECT` 中应写 `AVG(Grade)`；若直接写 `Grade`，在标准 SQL 中通常不合法。

   6. **ORDER BY 子句**

      | 排序方式 | SQL 格式 | 说明 |
      | --- | --- | --- |
      | 升序 | `ORDER BY 列名1, 列名n ASC;` | 按一个或多个属性列升序排列；若不说明次序，默认为升序 |
      | 降序 | `ORDER BY 列名1, 列名n DESC;` | 按一个或多个属性列降序排列 |

      例：查询 `SC` 表中 `Cno` 为 3 的学生的 `Sno` 和 `Grade`，结果按照 `Grade` 降序排列。
      ```sql
      SELECT Sno, Grade
      FROM SC
      WHERE Cno = '3'
      ORDER BY Grade DESC;
      ```
   7. **连接查询**
      1. **两表连接查询**
         1. 基本格式：
            ```sql
            SELECT 目标列表达式
            FROM 表名1, 表名2
            WHERE 表名1.列名1 比较运算符 表名2.列名2;
            ```
         2. 当列名在参与连接的各表中唯一时，可以省去表名前缀；若不同表中有同名属性列，则必须加表名前缀。
         3. 例：查询 `Student` 表中学生情况，以及 `SC` 表中他们对应的选课情况，并在一个查询结果中展示。
            ```sql
            SELECT Student.*, SC.*
            FROM Student, SC
            WHERE Student.Sno = SC.Sno;
            ```
            其中 `Sno` 作为连接字段，将两个表连接在一起。
         4. 若想获得自然连接效果，可以只选择全部需要的属性列，并去掉重复属性列：
            ```sql
            SELECT Student.Sno, Sname, Ssex, Sage, Sdept, Cno, Grade
            FROM Student, SC
            WHERE Student.Sno = SC.Sno;
            ```
         5. 例：在 `Student` 表和 `SC` 表中，查询选修了 2 号课程（`Cno='2'`）且成绩 `Grade` 在 90 分以上的学生学号 `Sno` 和姓名 `Sname`。
            ```sql
            SELECT Student.Sno, Sname
            FROM Student, SC
            WHERE Student.Sno = SC.Sno
              AND SC.Cno = '2'
              AND SC.Grade > 90;
            ```
            `AND` 用作多重条件连接。

      2. **单表连接查询（自连接）**
         1. 自连接是同一个表与自身连接，通常需要给同一个表起不同别名。
         2. 例：在 `Course` 表中查询先修课的先修课（即间接先修课），其中课程号为 `Cno`，先修课号为 `Cpno`。
            ```sql
            SELECT FIRST.Cno, SECOND.Cpno
            FROM Course FIRST, Course SECOND
            WHERE FIRST.Cpno = SECOND.Cno;
            ```
            这里 `FIRST` 和 `SECOND` 是同一个 `Course` 表的两个别名。

      3. **外连接查询**
         1. 外连接会将悬浮元组保留在结果关系中，没有属性值的位置填 `NULL`。

         | 外连接类型 | SQL 格式 | 含义 |
         | --- | --- | --- |
         | 左外连接 | `FROM 表名1 LEFT OUTER JOIN 表名2 ON(连接条件)` | 保留左表中的悬浮元组 |
         | 右外连接 | `FROM 表名1 RIGHT OUTER JOIN 表名2 ON(连接条件)` | 保留右表中的悬浮元组 |
         | 全外连接 | `FROM 表名1 FULL OUTER JOIN 表名2 ON(连接条件)` | 两侧悬浮元组都保留 |

         2. `ON` 也可以换成 `USING`，用于去掉结果中的重复列。
         3. **右外连接格式**：
            ```sql
            FROM 表名1 RIGHT OUTER JOIN 表名2 ON(连接条件);
            ```
         4. **左外连接例子**：以 `Student` 表为主体，列出每个学生的基本情况和选课情况，没选课的学生仍保留在结果中。
            ```sql
            SELECT Student.Sno, Sname, Ssex, Sage, Sdept, Cno, Grade
            FROM Student LEFT OUTER JOIN SC ON (Student.Sno = SC.Sno);
            ```
            注意：这个例子保留的是 `Student` 表中的悬浮元组，因此使用的是左外连接。

      4. **多表连接查询**
         1. 基本格式：
            ```sql
            SELECT 目标列表达式
            FROM 表名1, 表名2, 表名3
            WHERE 表名1.列名1 = 表名2.列名2
              AND 表名2.列名2 = 表名3.列名3;
            ```
         2. 多表连接一般是先进行两个表的连接操作，再将其连接结果与第三个表执行连接。
         3. 例：从 `Student`、`SC`、`Course` 表中查询每个学生的学号 `Sno`、姓名 `Sname`、课程名 `Cname` 和成绩 `Grade`。
            ```sql
            SELECT Student.Sno, Sname, Cname, Grade
            FROM Student, SC, Course
            WHERE Student.Sno = SC.Sno
              AND SC.Cno = Course.Cno;
            ```

   8. **嵌套查询**
      1. **查询块**：一个 `SELECT-FROM-WHERE` 语句称为一个查询块。
      2. **嵌套查询**：将一个查询块嵌套在另一个查询块的 `WHERE` 子句或 `HAVING` 子句中。
         1. 上层查询块称为外层查询或父查询。
         2. 下层查询块称为内层查询或子查询。
      3. **注意**：子查询的 `SELECT` 语句中不能使用 `ORDER BY` 子句，`ORDER BY` 子句只能对最终查询结果排序。
      4. **IN 子查询**
         1. 父查询与子查询之间用 `IN` 连接。
         2. 格式：
            ```sql
            WHERE 列名 IN (子查询);
            ```
         3. 例：查找与刘晨同一个专业的同学。
            1. 先查刘晨所在专业：
               ```sql
               SELECT Sdept
               FROM Student
               WHERE Sname = '刘晨';
               ```
            2. 再查询该专业的学生：
               ```sql
               SELECT Sno, Sname, Sdept
               FROM Student
               WHERE Sdept = 'CS';
               ```
            3. 合并为嵌套查询：
               ```sql
               SELECT Sno, Sname, Sdept
               FROM Student
               WHERE Sdept IN
                  (SELECT Sdept
                   FROM Student
                   WHERE Sname = '刘晨');
               ```
               本例的子查询条件不依赖于父查询，这类子查询称为**不相关子查询**。

      5. **比较运算符子查询**
         1. 父查询与子查询之间用比较运算符连接。
         2. 当用户能明确知道内层查询返回的是单个值时，可以用比较运算符连接。
         3. 格式：
            ```sql
            WHERE 列名 比较运算符 (子查询);
            ```
         4. 例：在 `SC` 表中，找出每个学生超过他自己选修课程平均成绩的课程号。
            ```sql
            SELECT Sno, Cno
            FROM SC x
            WHERE Grade >=
               (SELECT AVG(Grade)
                FROM SC y
                WHERE y.Sno = x.Sno);
            ```
            其中 `x` 是 `SC` 表的别名，也可以看作元组变量，用来表示 `SC` 的一个元组。
            本例的子查询条件依赖于父查询，这类子查询称为**相关子查询**，整个查询称为**相关嵌套查询**。

      6. **ANY/ALL 子查询**
         1. 有的系统中 `ANY` 可以用 `SOME` 代替。
         2. 格式：
            ```sql
            WHERE 列名 比较运算符 ANY/ALL (子查询);
            ```

         | 谓词 | 语义 | 与聚集函数或 `IN` 的等价转换 |
         | --- | --- | --- |
         | `> ANY` | 大于子查询结果中的某个值 | `> MIN` |
         | `> ALL` | 大于子查询结果中的所有值 | `> MAX` |
         | `< ANY` | 小于子查询结果中的某个值 | `< MAX` |
         | `< ALL` | 小于子查询结果中的所有值 | `< MIN` |
         | `>= ANY` | 大于等于子查询结果中的某个值 | `>= MIN` |
         | `>= ALL` | 大于等于子查询结果中的所有值 | `>= MAX` |
         | `<= ANY` | 小于等于子查询结果中的某个值 | `<= MAX` |
         | `<= ALL` | 小于等于子查询结果中的所有值 | `<= MIN` |
         | `= ANY` | 等于子查询结果中的某个值 | `IN` |
         | `= ALL` | 等于子查询结果中的所有值 | 通常无实际意义 |
         | `!= ANY` 或 `<> ANY` | 不等于子查询结果中的某个值 | 通常不直接替换 |
         | `!= ALL` 或 `<> ALL` | 不等于子查询结果中的任何值 | `NOT IN` |

         例：在 `Student` 表中，查询不是 `CS` 专业的学生中，比 `CS` 专业任意一个学生年纪小的学生姓名和年龄。
         ```sql
         SELECT Sname, Sage
         FROM Student
         WHERE Sage < ANY
            (SELECT Sage
             FROM Student
             WHERE Sdept = 'CS')
           AND Sdept <> 'CS';
         ```

      7. **EXISTS 子查询**
         1. `EXISTS` 表示存在量词，与之对应的是 `NOT EXISTS`。
         2. `EXISTS` 谓词的子查询不返回任何实际数据，只产生逻辑真值 `true` 或逻辑假值 `false`。
         3. `EXISTS` 引出的子查询目标列表达式通常都写 `*`。

         **逻辑蕴涵表达法：**
         1. 例：在 `SC` 表中查询至少选修了 1 号学生选修的全部课程 `Cno` 的学生学号 `Sno`。
         2. 逻辑分析：查询学号为 `x` 的学生，对所有课程 `y`，只要 1 号学生选修了课程 `y`，则 `x` 也选修了 `y`。
         3. 令 $p$ 表示“学生 1 号选修了课程 $y$”，令 $q$ 表示“学生 $x$ 选修了课程 $y$”，则上述查询可以表示为 $(\forall y)(p \to q)$。
         4. 通过等价转换：
            $$
            (\forall y)(p\to q)\equiv \neg(\exists y)(\neg(p\to q))\equiv \neg(\exists y)(\neg(\neg p\lor q))\equiv \neg(\exists y)(p\land\neg q)
            $$
            含义是：不存在这样的课程 $y$，学生 1 号选修了 $y$，而学生 `x` 没有选。
         5. SQL 表达式：
            ```sql
            SELECT DISTINCT Sno
            FROM SC SCX
            WHERE NOT EXISTS
               (SELECT *
                FROM SC SCY
                WHERE SCY.Sno = '1'
                  AND NOT EXISTS
                     (SELECT *
                      FROM SC SCZ
                      WHERE SCZ.Sno = SCX.Sno
                        AND SCZ.Cno = SCY.Cno));
            ```
         6. 例：基于 `SC` 表，查询选修了全部课程 `Course` 表中课程的学生姓名 `Sname`。
            1. 逻辑分析：令 $p$ 表示“课程 $x$ 被学生 $y$ 选修了”，则有 $(\forall x)p\equiv \neg(\exists x)(\neg p)$。
            2. 含义：查询不存在任何一门课程是学生 `y` 没有选修的。
            ```sql
            SELECT Sname
            FROM Student
            WHERE NOT EXISTS
               (SELECT *
                FROM Course
                WHERE NOT EXISTS
                   (SELECT *
                    FROM SC
                    WHERE Sno = Student.Sno
                      AND Cno = Course.Cno));
            ```
   9. **集合查询**
      1. 多个 `SELECT` 语句的结果可以进行集合的并、交、差操作。
      2. 注意：参加集合操作的各查询结果列数必须相同，对应项的数据类型也必须相同。
      3. 一般格式：
         ```sql
         SELECT 语句1
         UNION/INTERSECT/EXCEPT
         SELECT 语句2;
         ```

      | 集合操作 | 含义 | 注意点 |
      | --- | --- | --- |
      | `UNION` | 并操作 | 默认自动去掉重复元组；若要保留重复元组，使用 `UNION ALL` |
      | `INTERSECT` | 交操作 | 取两个查询结果共有的元组 |
      | `EXCEPT` | 差操作 | 取第一个查询结果中有、第二个查询结果中没有的元组 |

      4. **UNION 并操作**
         1. 例：在 `Student` 表中查询 `CS` 专业的学生和年龄不大于 19 岁的学生。
            ```sql
            SELECT *
            FROM Student
            WHERE Sdept = 'CS'
            UNION
            SELECT *
            FROM Student
            WHERE Sage <= 19;
            ```
         2. 例：在 `SC` 表中查询选修了课程 1 或课程 2 的学生。
            ```sql
            SELECT Sno
            FROM SC
            WHERE Cno = '1'
            UNION
            SELECT Sno
            FROM SC
            WHERE Cno = '2';
            ```
      5. **INTERSECT 交操作**
         1. 例：查询既选修了课程 1 又选修了课程 2 的学生。
            ```sql
            SELECT Sno
            FROM SC
            WHERE Cno = '1'
            INTERSECT
            SELECT Sno
            FROM SC
            WHERE Cno = '2';
            ```
      6. **EXCEPT 差操作**
         1. 例：查询 `CS` 专业的学生与年龄不大于 19 岁的学生的差集。
            ```sql
            SELECT *
            FROM Student
            WHERE Sdept = 'CS'
            EXCEPT
            SELECT *
            FROM Student
            WHERE Sage <= 19;
            ```

   10. **基于派生表的查询**
      1. 子查询出现在 `FROM` 子句时，子查询将生成临时的派生表，成为主查询的查询对象。
      2. 格式：
         ```sql
         FROM (子查询) AS 别名(属性列名1, 属性列名n)
         ```
      3. 如果子查询中没有聚集函数，派生表可以不指定属性列，子查询 `SELECT` 子句后面的列名为其默认属性。
      4. `AS` 可以省略，但必须为派生表指定一个别名。
      5. 例：找出每个学生超过他自己选修课程平均成绩的课程号。
         ```sql
         SELECT Sno, Cno
         FROM SC,
              (SELECT Sno, AVG(Grade)
               FROM SC
               GROUP BY Sno) AS Avg_sc(avg_sno, avg_grade)
         WHERE SC.Sno = Avg_sc.avg_sno
           AND SC.Grade >= Avg_sc.avg_grade;
         ```
      6. 例：查询所有选修了 1 号课程的学生姓名。
         ```sql
         SELECT Sname
         FROM Student,
              (SELECT Sno
               FROM SC
               WHERE Cno = '1') AS SC1
         WHERE Student.Sno = SC1.Sno;
         ```

3. **⭐INSERT/UPDATE/DELETE 语句**
   1. **插入数据：INSERT**
      1. **插入元组**
         1. 格式：
            ```sql
            INSERT
            INTO 表名 (列名1, 列名n)
            VALUES (常量1, 常量n);
            ```
         2. 字符串常量要用单引号 `'` 括起来。
      2. **插入示例**：假设现有 `TAB1` 表，共有 `C1` 到 `C4` 四列，其中 `C4` 列是字符串常量。

         | 情况 | SQL 示例 | 说明 |
         | --- | --- | --- |
         | 明确给出全部列名并赋值 | `INSERT INTO TAB1 (C1, C2, C3, C4) VALUES (1, 2, 3, '4');` | 插入数据包含全部属性列 |
         | 省略列名，直接插入全部列 | `INSERT INTO TAB1 VALUES (1, 2, 3, '4');` | 需要按表中列的定义顺序赋值 |
         | 指明部分列并赋值 | `INSERT INTO TAB1 (C1, C2, C3) VALUES (1, 2, 3);` | 未赋值的 `C4` 会被置为 `NULL` |
         | 省略列名但显式写 `NULL` | `INSERT INTO TAB1 VALUES (1, 2, 3, NULL);` | 必须明确将未赋值的属性列写为 `NULL` |

      3. 代码块写法：
         ```sql
         INSERT
         INTO TAB1 (C1, C2, C3, C4)
         VALUES (1, 2, 3, '4');
         ```
         ```sql
         INSERT
         INTO TAB1
         VALUES (1, 2, 3, '4');
         ```
         ```sql
         INSERT
         INTO TAB1 (C1, C2, C3)
         VALUES (1, 2, 3);
         ```
         ```sql
         INSERT
         INTO TAB1
         VALUES (1, 2, 3, NULL);
         ```
      4. 注意：若表定义中说明了 `NOT NULL`，但插入时不给该列赋值，则会出错。
      5. **插入子查询结果**
         1. 子查询嵌套在 `INSERT` 语句中，用于生成要插入的批量数据。
         2. 格式：
            ```sql
            INSERT
            INTO 表名 (属性列1, 属性列n)
            子查询;
            ```
         3. 例：假设已有 `TAB1` 表，并按 `C1` 分组求 `C2` 的平均值，存入 `TAB2` 表；其中 `TAB2` 的 `C1` 列存放 `C1`，`avg_C2` 列存放 `C2` 的均值。
            ```sql
            INSERT
            INTO TAB2 (C1, avg_C2)
            SELECT C1, AVG(C2)
            FROM TAB1
            GROUP BY C1;
            ```
   2. **修改数据：UPDATE**
      1. 格式：
         ```sql
         UPDATE 表名
         SET 列名1 = 表达式1, 列名n = 表达式n
         WHERE 条件;
         ```
      2. `WHERE` 用来指定要修改的元组；若省略 `WHERE`，表示修改表中的所有元组。
      3. **修改示例**

         | 情况 | SQL 示例 | 说明 |
         | --- | --- | --- |
         | 修改某一个元组的值 | `UPDATE TAB1 SET C4='0' WHERE C1=1;` | 只修改满足条件的元组 |
         | 修改多个元组的值 | `UPDATE TAB1 SET C3=C3+1;` | 省略 `WHERE`，修改表中所有元组 |
         | 带子查询的修改语句 | `UPDATE TAB1 SET C4='0' WHERE C1 IN (SELECT C1 FROM TAB2 WHERE avg_C2=2);` | 根据子查询结果筛选要修改的元组 |

      4. 代码块写法：
         ```sql
         UPDATE TAB1
         SET C4 = '0'
         WHERE C1 = 1;
         ```
         ```sql
         UPDATE TAB1
         SET C3 = C3 + 1;
         ```
         ```sql
         UPDATE TAB1
         SET C4 = '0'
         WHERE C1 IN
            (SELECT C1
             FROM TAB2
             WHERE avg_C2 = 2);
         ```
   3. **删除数据：DELETE**
      1. 格式：
         ```sql
         DELETE
         FROM 表名
         WHERE 条件;
         ```
      2. `WHERE` 用来指定要删除的元组；若省略 `WHERE`，表示删除表中的所有元组。
      3. 注意：`DELETE` 删除的是表中的数据，不是表的定义，表的定义仍在数据字典中。
      4. **删除示例**

         | 情况 | SQL 示例 | 说明 |
         | --- | --- | --- |
         | 删除某一个元组的值 | `DELETE FROM TAB1 WHERE C1=1;` | 删除满足条件的元组 |
         | 删除多个元组的值 | `DELETE FROM TAB1;` | 省略 `WHERE`，`TAB1` 将变成空表 |
         | 带子查询的删除语句 | `DELETE FROM TAB1 WHERE C1 IN (SELECT C1 FROM TAB2 WHERE avg_C2=2);` | 根据子查询结果筛选要删除的元组 |

      5. 代码块写法：
         ```sql
         DELETE
         FROM TAB1
         WHERE C1 = 1;
         ```
         ```sql
         DELETE
         FROM TAB1;
         ```
         ```sql
         DELETE
         FROM TAB1
         WHERE C1 IN
            (SELECT C1
             FROM TAB2
             WHERE avg_C2 = 2);
         ```

4. **⭐VIEW 视图**
   1. **定义视图：CREATE VIEW**
      1. 格式：
         ```sql
         CREATE VIEW 视图名 (列名1, 列名n)
         AS 子查询
         WITH CHECK OPTION;
         ```
      2. 若省略视图列名，则该视图的列名由子查询中 `SELECT` 的目标列字段组成。
      3. `WITH CHECK OPTION`：表示对视图进行增删改时，必须满足子查询中的条件表达式。
      4. 以下情况必须明确指定组成视图的所有列名：

         | 情况 | 原因 |
         | --- | --- |
         | 目标列不是单纯的列名，而是聚集函数或列表达式 | 需要给计算结果指定列名 |
         | 多表连接时选出了几个同名列作为视图字段 | 避免视图中出现同名字段 |
         | 需要在视图中为某个列启用新的、更合适的名字 | 提高视图字段含义的清晰度 |

      5. **行列子集视图**：由单个基本表导出，只去掉了基本表的某些行和某些列，但保留了主码。
      6. 若某些视图由多个基本表导出，或缺少基本表的全部属性列，视图与基本表的各列不是一一对应的，修改基本表结构时，基本表和视图的映像关系可能被破坏。这种情况下，最好在修改基本表后删除该视图，然后重建该视图。
   2. **CREATE VIEW 示例**
      1. 建立 `TAB1` 的视图。
         ```sql
         CREATE VIEW V_TAB1
         AS
         SELECT C1, C2, C3, C4
         FROM TAB1
         WHERE C1 = 1;
         ```
      2. 建立 `C4` 为 4 时 `TAB1` 的视图，并保证以后每次增删改时都满足 `C4` 为 4 的条件。
         ```sql
         CREATE VIEW V_TAB2
         AS
         SELECT C1, C2, C3, C4
         FROM TAB1
         WHERE C4 = '4'
         WITH CHECK OPTION;
         ```
      3. 建立在一个或多个已定义好的视图上。
         ```sql
         CREATE VIEW V_TAB3
         AS
         SELECT C1, C2, C3
         FROM V_TAB1
         WHERE C2 = 2;
         ```
      4. 为减少冗余数据，基本表中一般只存放基本数据；需要使用计算得到的派生数据时，可以设置在视图中的派生属性列上，也称为虚拟列。带虚拟列的视图也称为带表达式的视图。
         ```sql
         CREATE VIEW V_TAB4(C1, new_C2)
         AS
         SELECT C1, 10 + C2
         FROM TAB1;
         ```
      5. 分组视图：带有聚集函数和 `GROUP BY` 子句。
         ```sql
         CREATE VIEW V_TAB5(C1, avg_C2)
         AS
         SELECT C1, AVG(C2)
         FROM TAB1
         GROUP BY C1;
         ```
   3. **删除视图：DROP VIEW**
      1. 格式：
         ```sql
         DROP VIEW 视图名 CASCADE;
         ```
      2. 若使用 `CASCADE` 级联删除语句，则会把该视图导出的所有视图一并删除。
      3. 例：删除 `V_TAB2`。
         ```sql
         DROP VIEW V_TAB2;
         ```
      4. 例：删除 `V_TAB1`，并将由 `V_TAB1` 导出的 `V_TAB3` 也一并删除。
         ```sql
         DROP VIEW V_TAB1 CASCADE;
         ```
   4. **查询视图和更新视图**
      1. 视图定义后，对视图进行查询和更新的语句、语法与基本表相同。
      2. 视图的查询与更新最终都会转换为对基本表的查询和更新，这一过程称为**视图消解**。
      3. 一般来说，行列子集视图的查询和更新都可以顺利转换，其他视图不一定能转换。

5. **⭐空值**
   1. **判断一个属性是否为空值**

      | 判断 | SQL 格式 |
      | --- | --- |
      | 判断为空 | `属性 IS NULL` |
      | 判断非空 | `属性 IS NOT NULL` |

   2. **空值的运算**
      1. 空值与另一个值的算术运算结果为空值。
      2. 空值与另一个值的比较运算结果为 `UNKNOWN`。
      3. 在查询语句中，只有使 `WHERE` 和 `HAVING` 子句的选择条件为 `TRUE` 的元组才会被选出作为输出结果，即不包括 `UNKNOWN` 的情况。

# 事务调度
1. **⭐事务调度的准则**
   1. **一组事务的调度必须保证**
      1. 包含所有事务的操作指令。
      2. 一个事务内部的指令顺序必须保持不变。
   2. **并行事务调度必须保证**
      1. 并行调度必须是可串行化的。
      2. 判断方法：将所有可能的串行调度结果推演一遍，再对某个具体的并行调度执行一遍，看其结果是否能与某个串行调度的结果相同。
   3. **判断可串行化的充分条件：冲突可串行化**
      1. 冲突可串行化一定是可串行化调度，但可串行化调度不一定是冲突可串行化。
      2. **冲突操作**

         | 冲突类型 | 含义 |
         | --- | --- |
         | 读写冲突 | 不同事务对同一数据分别进行读和写 |
         | 写写冲突 | 不同事务对同一数据分别进行写和写 |

      3. **冲突可串行化调度**：不交换不同事务的冲突操作次序，也不交换同一事务内部两个操作的次序；但可以交换不同事务对不同数据的各种操作次序，也可以交换不同事务对同一数据的读取操作次序。
2. **⭐封锁**
   1. **X 锁（写锁）**

      | 内容 | 说明 |
      | --- | --- |
      | 含义 | 某事务对数据对象上 X 锁后，可以读取和修改该数据对象 |
      | 对其他事务的限制 | 其他事务不可再对该数据对象添加锁 |
      | 表示方法 | 上锁 `Xlock()`；释放锁 `Unlock()` |

   2. **S 锁（读锁）**

      | 内容 | 说明 |
      | --- | --- |
      | 含义 | 某事务对数据对象上 S 锁后，可以读取但不能修改该数据对象 |
      | 对其他事务的限制 | 其他事务可以对该数据对象添加 S 锁，但不能添加 X 锁 |
      | 表示方法 | 上锁 `Slock()`；释放锁 `Unlock()` |

   3. **封锁协议**

      | 封锁协议 | 规则 | 能防止的问题 |
      | --- | --- | --- |
      | 一级封锁协议 | 写前加写锁，事务结束释放写锁 | 可防止丢失修改 |
      | 二级封锁协议 | 写前加写锁，读前加读锁；读完释放读锁，事务结束释放写锁 | 可防止丢失修改和读脏数据 |
      | 三级封锁协议 | 写前加写锁，读前加读锁，事务结束释放所有锁 | 可防止丢失修改、读脏数据和不可重复读 |

      1. **一级封锁协议**：写前加写锁，事务结束释放写锁；可防止丢失修改。
      2. **二级封锁协议**：写前加写锁，读前加读锁，读完释放读锁，事务结束释放写锁；可防止丢失修改和读脏数据。
      3. **三级封锁协议**：常用于支持一致性维护；写前加写锁，读前加读锁，事务结束释放各锁；可防止丢失修改、读脏数据和不可重复读。
      4. 如果所有事务都遵循三级封锁协议，由于其隔离级别高，那么这些事务无论怎样交叉并行，都是可串行化的调度。

   4. **两段锁协议（2PL）**
      1. 三级封锁协议可以保证并发操作的正确性，但由于其太过严格，对并发度有负面影响。
      2. 三级封锁协议实际上是两段锁协议的特例，是更严格的两段锁协议。
      3. **两段锁协议要求**：事务在对任何数据进行读写前，必须获得该数据的封锁；当事务释放任何一个封锁后，不可再获得任何其他封锁。
      4. 事务遵循两段锁协议是可串行化的充分条件。
      5. 遵循两段锁协议仍然可能发生死锁。
3. **⭐例题：并发调度与两段锁协议**
   1. **题目条件**
      1. 初始值：$A=20$，$B=10$。
      2. `X`、`Y` 是事务内部的局部变量。
      3. `X = Read(A)` 表示读取数据项 `A` 的值到局部变量 `X` 中；`Write(A, X)` 表示将局部变量 `X` 的值写入数据项 `A` 中。
   2. **原并发调度**

      | 步骤 | $T_1$ | $T_2$ |
      | --- | --- | --- |
      | 1 | `X = Read(A);` |  |
      | 2 | `Y = Read(B);` |  |
      | 3 |  | `X = Read(A);` |
      | 4 |  | `X = X - 10;` |
      | 5 | `X = X + Y;` |  |
      | 6 |  | `Write(A, X);` |
      | 7 |  | `Y = Read(B);` |
      | 8 | `Write(A, X);` |  |
      | 9 |  | `Y = Y * X;` |
      | 10 |  | `Write(B, Y);` |

   3. **原调度执行结果**
      1. $T_1$：`X=20`，`Y=10`，所以 `X=X+Y=30`，最后写入 `A=30`。
      2. $T_2$：先读到 `A=20`，再令 `X=X-10=10`，写入过一次 `A=10`；随后读到 `B=10`，令 `Y=Y*X=100`，最后写入 `B=100`。
      3. 因此原调度最终结果为：$A=30$，$B=100$。
   4. **判断是否正确**
      1. 若串行执行 $T_1 \to T_2$，最终结果为：$A=20$，$B=200$。
      2. 若串行执行 $T_2 \to T_1$，最终结果为：$A=110$，$B=100$。
      3. 原并发调度结果 $A=30$，$B=100$ 与任一串行结果都不同，因此该调度不是可串行化调度，是不正确的。
      4. 原因：$T_2$ 对 `A` 的修改被 $T_1$ 基于旧值计算出的 `A=30` 覆盖，出现了丢失修改问题。
   5. **加入 `Xlock()` 和 `Unlock()` 后的两段锁调度**
      要求每个事务首条语句的请求时刻与图示调度一致，因此 $T_1$ 先申请 `A` 的锁，$T_2$ 在原来首次读 `A` 的位置申请 `A` 的锁并等待。

      | 步骤 | $T_1$ | $T_2$ |
      | --- | --- | --- |
      | 1 | `Xlock(A);` |  |
      | 2 | `X = Read(A);` |  |
      | 3 | `Xlock(B);` |  |
      | 4 | `Y = Read(B);` |  |
      | 5 |  | `Xlock(A);`（等待） |
      | 6 | `X = X + Y;` |  |
      | 7 | `Write(A, X);` |  |
      | 8 | `Unlock(A);` |  |
      | 9 | `Unlock(B);` |  |
      | 10 |  | `Xlock(A);`（获得） |
      | 11 |  | `X = Read(A);` |
      | 12 |  | `X = X - 10;` |
      | 13 |  | `Write(A, X);` |
      | 14 |  | `Xlock(B);` |
      | 15 |  | `Y = Read(B);` |
      | 16 |  | `Y = Y * X;` |
      | 17 |  | `Write(B, Y);` |
      | 18 |  | `Unlock(A);` |
      | 19 |  | `Unlock(B);` |

   6. **加锁后调度结果**
      1. 加锁后实际等价于串行顺序 $T_1 \to T_2$。
      2. $T_1$ 执行后：$A=30$，$B=10$。
      3. $T_2$ 执行时读到 `A=30`，令 `X=20`，写入 `A=20`；再读 `B=10`，令 `Y=200`，写入 `B=200`。
      4. 因此新调度最终结果为：$A=20$，$B=200$。

# E-R 图
1. **⭐画 E-R 图（概念结构设计）**
   1. **基本图形**
      ```mermaid
      flowchart LR
         E[实体]
         R{联系}
         A([属性])
      ```

      | 图形 | 含义 | 说明 |
      | --- | --- | --- |
      | 矩形 | 实体 | 表示现实世界中可以区别的对象 |
      | 菱形 | 联系 | 表示实体之间的联系 |
      | 椭圆 | 属性 | 实体和联系都可以具有属性 |

   2. **实体和联系都可以有属性**
      ```mermaid
      flowchart LR
         EA([实体属性]) --- E[实体]
         RA([联系属性]) --- R{联系}
      ```

   3. **联系类型**
      1. `1:1` 联系：1 个 `A` 对应 1 个 `B`。
         ```mermaid
         flowchart LR
            A[A] ---|1| R{联系}
            R ---|1| B[B]
         ```
      2. `1:n` 联系：1 个 `A` 对应 n 个 `B`。
         ```mermaid
         flowchart LR
            A[A] ---|1| R{联系}
            R ---|n| B[B]
         ```
      3. `m:n` 联系：n 个 `A` 对应 m 个 `B`。
         ```mermaid
         flowchart LR
            A[A] ---|n| R{联系}
            R ---|m| B[B]
         ```

      | 联系类型 | 读法 | 记忆 |
      | --- | --- | --- |
      | `1:1` | 一对一 | 1 个 `A` 最多对应 1 个 `B`，1 个 `B` 最多对应 1 个 `A` |
      | `1:n` | 一对多 | 1 个 `A` 可以对应多个 `B`，每个 `B` 通常只对应 1 个 `A` |
      | `m:n` | 多对多 | 多个 `A` 可以对应多个 `B` |

2. **⭐E-R 图转换为关系模型（逻辑结构设计）**
   1. **转换步骤**
      1. 第一步：将各个实体的名字转换为各个关系模式的名字。
      2. 第二步：实体的属性就是关系的属性，实体的码就是关系的码。
      3. 第三步：实体间联系的转换。

   2. **实体间联系的转换规则**

      | 联系类型 | 转换规则 | 记忆 |
      | --- | --- | --- |
      | `1:1` 联系 | 在任意一方加入对方的主码，并设为其外码；同时加入联系本身的属性 | 任意一边合并都可以 |
      | `1:n` 联系 | 将 `1` 方的主码加入 `n` 方作为外码；同时将联系的属性加入 `n` 方 | 外码放在多端 |
      | `m:n` 联系 | 将联系本身转换为一个关系模式；将联系双方的主码加入其中并设为码；同时加入联系的属性 | 多对多单独成表 |

   3. **速记口诀**
      1. 实体先成表：实体名变关系名，实体属性变关系属性，实体码变关系码。
      2. `1:1`：任选一方并入另一方。
      3. `1:n`：把 `1` 方主码放到 `n` 方。
      4. `m:n`：联系单独成表，双方主码共同做码。

3. **⭐例题：学校 E-R 图**
   1. **题目关键信息**
      1. 学校有若干系。
      2. 每个系有若干班级和教研室。
      3. 每个教研室有若干教员。
      4. 教授和副教授每人各带若干研究生。
      5. 每个班有若干学生。
      6. 每个学生选修若干课程，每门课程可由若干学生选修。

   2. **实体集**

      | 实体 | 含义 |
      | --- | --- |
      | 学校 | 学校整体 |
      | 系 | 学校下设的院系 |
      | 班级 | 系下设的班级 |
      | 教研室 | 系下设的教研室 |
      | 教员 | 教研室中的教师；其中教授、副教授可以指导研究生 |
      | 学生 | 班级中的学生，也包括被指导的研究生 |
      | 课程 | 学生可选修的课程 |

   3. **联系及基数**

      | 联系 | 参与实体 | 基数 | 说明 |
      | --- | --- | --- | --- |
      | 校-系 | 学校、系 | `1:m` | 一个学校有多个系，一个系属于一个学校 |
      | 系-班 | 系、班级 | `1:n` | 一个系有多个班级，一个班级属于一个系 |
      | 系-室 | 系、教研室 | `1:n` | 一个系有多个教研室，一个教研室属于一个系 |
      | 班-生 | 班级、学生 | `1:n` | 一个班级有多个学生，一个学生属于一个班级 |
      | 室-教 | 教研室、教员 | `1:n` | 一个教研室有多个教员，一个教员属于一个教研室 |
      | 指导 | 教员、学生 | `1:n` | 一个教授/副教授可以指导多个研究生 |
      | 选修 | 学生、课程 | `m:n` | 一个学生可选多门课，一门课可被多个学生选修 |

   4. **E-R 图**
      ```mermaid
      flowchart TD
         School[学校]
         Dept[系]
         Class[班级]
         Room[教研室]
         Teacher[教员]
         Student[学生]
         Course[课程]

         R1{校-系}
         R2{系-班}
         R3{系-室}
         R4{班-生}
         R5{室-教}
         R6{指导}
         R7{选修}

         School ---|1| R1
         R1 ---|m| Dept

         Dept ---|1| R2
         R2 ---|n| Class

         Dept ---|1| R3
         R3 ---|n| Room

         Class ---|1| R4
         R4 ---|n| Student

         Room ---|1| R5
         R5 ---|n| Teacher

         Teacher ---|1| R6
         R6 ---|n| Student

         Student ---|m| R7
         R7 ---|n| Course
      ```

   5. **E-R 图转换后的关系模型**
      说明：下划线属性表示该关系的**主码属性**。

      | 关系模式 | 主码 | 外码 / 来源说明 |
      | --- | --- | --- |
      | 系（<u>系编号</u>，系名，学校名） | 系编号 | `学校名` 来自 `校-系` 的 `1:m` 联系，加入 `m` 端“系” |
      | 班级（<u>班级编号</u>，班级名，系编号） | 班级编号 | `系编号` 来自 `系-班` 的 `1:n` 联系，加入 `n` 端“班级” |
      | 教研室（<u>教研室编号</u>，教研室，系编号） | 教研室编号 | `系编号` 来自 `系-室` 的 `1:n` 联系，加入 `n` 端“教研室” |
      | 学生（<u>学号</u>，姓名，学历，班级编号，导师职工号） | 学号 | `班级编号` 来自 `班-生`；`导师职工号` 来自 `指导` 联系 |
      | 课程（<u>课程编号</u>，课程名） | 课程编号 | 实体“课程”直接转换 |
      | 教员（<u>职工号</u>，姓名，职称，教研室编号） | 职工号 | `教研室编号` 来自 `室-教` 的 `1:n` 联系，加入 `n` 端“教员” |
      | 选课（<u>学号</u>，<u>课程编号</u>，成绩） | 学号 + 课程编号 | `选修` 是 `m:n` 联系，单独转换为关系模式；`成绩` 是联系属性 |

      1. `1:n` 联系不用单独成表，一般把 `1` 方主码加入 `n` 方作为外码。
      2. `m:n` 联系必须单独成表，双方主码共同作为新关系的主码。
      3. 本题中 `选修` 是多对多联系，所以转换为 `选课（学号，课程编号，成绩）`。

   6. **考试写法提醒**
      1. 先找实体：名词通常优先考虑实体，例如学校、系、班级、学生、课程等。
      2. 再找联系：动词或所属关系通常优先考虑联系，例如属于、包含、选修、指导等。
      3. 最后标基数：看到“若干”通常意味着多端；“每个 X 有若干 Y”通常是 `X:Y = 1:n`。

# 查询优化

1. **⭐文件组织方式与记录组织**
   1. **数据库访问的核心矛盾**
      数据库数据主要存放在磁盘等外存中，DBMS 执行查询时需要把数据块调入内存处理。查询优化的主要目标之一就是**减少磁盘 I/O 次数**。

   2. **文件组织方法**

      | 文件组织方式 | 存储特点 | 优点 | 缺点 / 适用场景 |
      | --- | --- | --- | --- |
      | **堆文件** | 记录按插入顺序存放，无固定排序 | 插入简单、速度快 | 查询通常要顺序扫描，适合插入多、查询少 |
      | **顺序文件** | 按某个排序码有序存放 | 按排序码范围查询效率高 | 插入、删除可能需要移动记录，维护成本较高 |
      | **索引文件** | 另建索引项，通过索引定位数据记录 | 条件查询快 | 占额外空间，更新数据时要维护索引 |
      | **散列文件** | 用散列函数把记录分配到桶中 | 等值查询很快 | 不适合范围查询，可能产生冲突和溢出桶 |

      **速记**：堆文件重插入，顺序文件重范围，索引文件重条件查找，散列文件重等值查找。

2. **⭐索引和散列**
   1. **索引的基本概念**
      索引类似书本目录，由**查找码值 + 指向记录的指针**组成。建立索引可以减少查找数据时扫描的记录数，但会增加存储空间和更新维护成本。

   2. **常见索引类型**

      | 分类 | 含义 | 记忆 |
      | --- | --- | --- |
      | **主索引 / 聚簇索引** | 数据文件按查找码有序组织，索引顺序与记录物理顺序一致 | 一张表通常只能有一个 |
      | **辅助索引 / 非聚簇索引** | 索引顺序与记录物理顺序不一定一致 | 可以有多个 |
      | **稠密索引** | 每个查找码值都有索引项 | 查找快，占空间多 |
      | **稀疏索引** | 只为部分查找码值建索引项，通常每个数据块一个 | 占空间少，但定位后还要块内查找 |
      | **多级索引** | 对索引再建索引 | 适合索引文件很大时使用 |

   3. **B+ 树索引**
      1. B+ 树是数据库中常用的动态索引结构。
      2. 非叶结点只起索引作用，叶结点保存查找码及指针。
      3. 所有叶结点在同一层，树高较低，查找路径稳定。
      4. 叶结点之间通常按查找码顺序链接，既适合**等值查询**，也适合**范围查询**。
      5. 插入、删除时通过结点分裂、合并或重分配保持平衡。

   4. **散列索引**
      散列通过散列函数 `H(key)` 直接计算记录所在桶号。

      | 类型 | 特点 | 适用 |
      | --- | --- | --- |
      | **静态散列** | 桶数预先固定，冲突时使用溢出桶 | 数据规模较稳定 |
      | **动态散列** | 桶数可随数据增长而扩充 | 数据规模变化较大 |

      **散列优缺点**：等值查询快；范围查询差；散列函数要尽量均匀，否则容易产生冲突。

3. **⭐查询处理**
   1. **查询处理步骤**

      | 步骤 | 含义 |
      | --- | --- |
      | ① 查询分析和检查 | 对 SQL 做词法、语法、语义检查 |
      | ② 转换为内部表示 | 通常转换为关系代数表达式或查询树 |
      | ③ 查询优化 | 选择代价较小的等价执行方案 |
      | ④ 生成执行计划 | 确定访问路径和具体算法 |
      | ⑤ 执行并返回结果 | 调用存储管理器读写数据，输出查询结果 |

   2. **选择运算的实现**

      | 方法 | 思路 | 适用情况 |
      | --- | --- | --- |
      | **全表扫描** | 顺序读取表中所有记录，逐条判断条件 | 小表、无索引、结果占比高 |
      | **索引扫描** | 先通过索引定位满足条件的记录，再访问数据记录 | 大表、选择条件上有合适索引、结果占比低 |

   3. **连接运算的实现**

      | 方法 | 思路 | 特点 |
      | --- | --- | --- |
      | **嵌套循环连接** | 外层表每一行与内层表逐行比较 | 最基本，适用广但大表代价高 |
      | **排序-合并连接** | 两表按连接属性排序后同步扫描 | 适合已排序或可高效排序的表 |
      | **索引连接** | 利用内层表连接属性上的索引查找匹配记录 | 外层表较小、内层表有索引时较好 |
      | **散列连接** | 按连接属性散列分桶，再在桶内匹配 | 适合等值连接 |

4. **⭐查询优化**
   1. **为什么需要查询优化**
      SQL 是非过程化语言，用户只说明“查什么”，不说明“怎么查”。同一个查询可以有多种等价执行方法，DBMS 需要选择代价较小的方案。

   2. **查询优化的两类方法**

      | 类型 | 含义 | 例子 |
      | --- | --- | --- |
      | **代数优化 / 逻辑优化** | 利用关系代数等价变换改写查询表达式 | 选择下推、投影下推、笛卡儿积转连接 |
      | **物理优化** | 选择具体存取路径和执行算法 | 用索引扫描还是全表扫描；用哪种连接算法 |

   3. **常用优化策略**
      1. **选择运算尽早执行**：先筛行，减少中间结果。
      2. **投影运算尽早执行**：先筛列，减少中间结果宽度。
      3. **把选择与笛卡儿积合并为连接**：避免生成巨大笛卡儿积。
      4. **优先利用索引或排序结果**：减少扫描和比较代价。
      5. **公共子表达式可复用**：避免重复计算。

   4. **常用等价变换规则**

      | 规则 | 公式 / 含义 |
      | --- | --- |
      | 选择串接 | $\sigma_{F_1}(\sigma_{F_2}(E)) \equiv \sigma_{F_1 \land F_2}(E)$ |
      | 投影串接 | $\Pi_{A}(\Pi_{A,B}(E)) \equiv \Pi_A(E)$ |
      | 选择下推 | 若条件只涉及某个关系，应尽量推到该关系上先执行 |
      | 投影下推 | 只保留后续查询和连接需要的属性 |
      | 连接交换 | $E_1 \Join E_2 \equiv E_2 \Join E_1$ |
      | 笛卡儿积 + 选择 | $\sigma_{R.a=S.a}(R \times S) \equiv R \Join_{R.a=S.a} S$ |

5. **⭐关系代数语法树表示代数优化过程**
   1. **例题：查询设计所中参加 201901 项目的人员姓名和职称**

      ```sql
      SELECT Ename, Title
      FROM Employee, Item_Emp, Department
      WHERE Employee.Eno = Item_Emp.Eno
        AND Employee.Dno = Department.Dno
        AND Dname = '设计所'
        AND Ino = '201901';
      ```

   2. **初始关系代数表达式**

      $$
      \Pi_{Ename,Title}
      (\sigma_{Employee.Eno=Item\_Emp.Eno \land Employee.Dno=Department.Dno \land Dname='设计所' \land Ino='201901'}
      (Employee \times Item\_Emp \times Department))
      $$

      问题：先做三表笛卡儿积，会产生很大的中间结果。

   3. **初始语法树**

      ```mermaid
      flowchart TD
         A["Π Ename, Title"] --> B["σ 连接条件 AND Dname='设计所' AND Ino='201901'"]
         B --> C["×"]
         C --> D["×"]
         D --> E["Employee"]
         D --> F["Item_Emp"]
         C --> G["Department"]
      ```

   4. **代数优化步骤**
      1. **拆分选择条件**：把一个复杂 `σ` 拆成多个简单 `σ`。
      2. **选择下推**：`Dname='设计所'` 推到 `Department`，`Ino='201901'` 推到 `Item_Emp`。
      3. **笛卡儿积 + 连接条件转连接**：把 `Employee × Item_Emp × Department` 改为连接运算。
      4. **投影下推**：每个关系只保留后面需要的列，如 `Employee` 保留 `Eno, Dno, Ename, Title`。

   5. **优化后语法树**

      ```mermaid
      flowchart TD
         A["Π Ename, Title"] --> B["⋈ Employee.Dno = Department.Dno"]
         B --> C["⋈ Employee.Eno = Item_Emp.Eno"]
         B --> D["Π Dno (σ Dname='设计所'(Department))"]
         C --> E["Π Eno,Dno,Ename,Title (Employee)"]
         C --> F["Π Eno (σ Ino='201901'(Item_Emp))"]
      ```

   6. **优化效果**
      1. 先筛选 `Department` 和 `Item_Emp`，参与连接的行数减少。
      2. 先投影，只保留必要列，中间结果宽度减少。
      3. 用连接代替笛卡儿积后再筛选，避免产生大量无用组合。

6. **本章速记**

   | 考点 | 一句话 |
   | --- | --- |
   | 文件组织 | 堆重插入，顺序重范围，索引重条件，散列重等值 |
   | 索引 | 目录思想：查找码 + 指针，加快查询但增加维护成本 |
   | B+ 树 | 平衡、多路、叶子有序链接，等值和范围查询都适合 |
   | 散列 | 等值查询快，范围查询差，关键看散列函数是否均匀 |
   | 查询处理 | SQL → 语法检查 → 查询树 → 优化 → 执行计划 → 结果 |
   | 查询优化 | 选择投影尽早下推，笛卡儿积尽量变连接 |
   | 语法树优化 | 先拆条件，再下推选择/投影，最后用连接减少中间结果 |

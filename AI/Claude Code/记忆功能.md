# 四层记忆存储

## 指令文件(CLAUDE.md)—人手动编写的记忆

唯一由人手动编写的记忆，告诉 Claude 项目的规则与约束

- **管理员全局**：/etc/calude-code/CLAUDE.md —IT管理员设定的全局规则
- **用户全局**：\~/.claude/CLAUDE.md —个人偏好（跨项目生效）
- **项目级**： CLAUDE.md、.claude/CLAUDE.MD、.claude/rules/\*.md —随着Git提交，团队共享
- **本地私有**：CLAUDE.local.md —不提交Git，个人使用

支持 @include 指令引用其他文件，上限40000字符。建议写法：**短，有主见，可操作**

## 对话上下文(messages[]) — 短期记忆

## Session Memory — 当前会话的结构化笔记

一个后台子代理(Fork Agent) 在工作的同时维护结构化笔记文件。

用于进行独立的信息提取，内容包括：任务标题和描述、当前状态、涉及的文件列表、遇到的错误和解决方案、经验教训、工作日志。

**触发条件**：对话token超过 10K 后启动，之后每 5K tokens 更新一次。后台执行，不阻塞工作。

价值：

- /resume 恢复会话的时候，用于快速恢复上下文，无需重新喂全部历史
- 上下文压缩时可以作为摘要来源
- /skilllify 可以将其转化为可服用的SKILL.md文件

## 持久记忆（Memdir）—跨会话的长期记忆

最核心的长期记忆，位于 \~/.claude/memory/。存 4 类信息：

- **user**：用户角色、偏好、技能水平。例：“是前端开发工程师”
- **feedback**：用户对 AI 的纠正和确认。例：“上次做了兜底处理，被用户要求删除兜底”
- **project**：项目目标、决策、截止日期。例：”6月10日要完成记忆存储架构“
- **reference**：外部系统指针。例：”项目文档在飞书：xxx“

这里故意**不存代码事实**

# 上下文的组装

## 指令文件的加载

所有层级的 [CLAUDE.md](http://CLAUDE.md) 都会被拼接在一起注入，不会相互覆盖。

拼接顺序为：管理员全局→用户全局→项目级→本地私有。

**设计理念：**注释中写道：”Files are loaded inreverse order of priority, i.e. the latest files are highest priority with the model paying more attention to them.”利用的是 LLM 的特性：模型对 prompt 末尾的内容天然给予更高的注意力（recency bias）。因此排在后面的文件在实践中“优先级更高”。

**举例**：如果管理员规则说“写出所有参数的校验”，但项目级 [CLAUDE.md](http://CLAUDE.md) 说”只保留部分必要的校验“，模型会倾向于后者。

## 持久记忆加载

1. [MEMORY.md](http://MEMORY.md) 前200行**每次会话**加载到 system prompt
2. 用 **Sonnet** （小模型）扫描记忆文件的标题和 frontmatter 描述，选出**最多 5 条**最相关的，作为  relevant_memories 附件注入上下文
3. 排除本次会话已召回过的记忆（避免重复）
4. 超过 1 天的记忆自动加过期警告：”这条记忆是N天前的…关于代码性为的声明可能已过时，请先验证。“
5. 会话总预算 60KB

**设计理念**：

Q：为什么不用RAG？

A：RAG有多个问题：代码修改后索引不同步，索引的权限管控复杂。最终发现 **agentic search (模型自主调用 glob + grep) 全面优于RAG** — 原话为”agentic search just outperformed everything,and when I say agentic search,this is a fancy word for glob and grep.”

## 压缩

Claude Code 设计了一套 **5 层压缩流水线**，在每轮查询循环中按照固定顺序依次执行，每层都有独立的触发条件—不是每层每次都触发，而是逐层检查，按需执行，越往后压缩程度越大。

```Mermaid
flowchart TD
    A["<b>第一层：Tool Result Budget</b><br>每轮都跑<br>限制单条信息体积"] --> B["<b>第二层：Microcompact</b><br>清理旧工具结果内容"]
    B --> C["<b>第三层：Auto-compact</b><br>触发 LLM 全量摘要<br>（核心层）"]
    C --> D["<b>第四层：Blocking Limit</b><br>硬拒绝<br>压缩失败则报错"]
    D --> E["<b>第五层：Reactive Compact</b><br>API 返回 413 后<br>兜底重试"]
```

### 第一层：Tool Result Budget（工具结果预算）

每轮都执行，比如说一轮对话让 Claude 跑了多个并行的工具调用，每个都返回几十KB，单条消息结果超过了 **200K 字符**的预算上限，这时候系统会把最大的工具结果**存到磁盘**，原位替换成约 2KB 的预览+文件路径。模型后续需要完整内容时，可以用 Read 工具去读取磁盘上的完整文件。大部分时候 2KB 的预览足够使用，完整的文件可能永远不会被读取。

**设计理念：**为了压缩工具，如果工具调用结果返回内容太多，就只给索引，按需加载。类似于SKILL的思想，**渐进式披露（Progressive Disclosure）**的思想—先给模型足够判断的预览，让模型自己决定要不要全量加载。

### 第二层：Microcompact（工具结果清理）

两个策略，按照优先级尝试，命中一个就停止，如果都不命中，则本层跳过。

1. **基于时间的 Microcompact**：如果距离上次的LLM回复超过了 **60分钟**（即缓存已过期），直接将旧的工具结果内容替换为 [Old tool result content cleared] ，只保留最近5条。适用的工具类型：Read，Bash，Grep，Glob，webSsearch，Edit，Write。因为此时缓存已失效，直接修改内容无损伤。
2. **基于缓存编辑的 Microcompact**（Anthropic 内部）：客户端消息完全不改，而是额外发送一条 cache_edits 指令，告诉服务器“在你缓存的副本里删掉这些工具结果”—prompt前缀不变，缓存继续有效。客户端本地的消息数组保持完整（用于 session 持久化、resume 等），但模型实际看到的上下文已经被压缩了。

**设计理念：Prompt Cache**机制：Claude Code的每次API调用都会把完整对话历史（system prompt + 所有消息）重新发给服务端。服务端会**缓存这个输入**（ TTL 约 60 分钟），后续请求如果前缀相同就直接复用，不需要从头处理。基于这个机制，这一层解决的问题就是：**怎么清理旧的工具结果，同时尽量不破坏这个缓存输入**。

### 第三层：Auto-compact（自动全量压缩）—核心层

**触发阈值**（以 200K 上下文窗口为例）：

```Markdown
有效窗口 = 200,000 -min(maxOutputTokens , 20,000) = 180,000
自动压缩阈值 = 180,000 - 13,000 = 167,000 tokens
```

即当前对话 token 超过 **167K** 的时候自动触发。

出发后有两个子策略，先尝试轻量的，不行再用重量的。

1. **Session Memory Compact（轻量）**：用已经由后台提取好的 Session Memory 文件作为摘要，保留最近 **10K\~40K tokens** 的原始对话（至少保留 5 条包含文本的消息）。**不需要额外的 LLM 调用**。
2. **Full LLM Compact（重量）**：把整段对话发给 LLM 做结构化摘要。摘要输出上限 **20K tokens**，包含9个固定段落：

   - 核心请求（Primary Request）
   - 关键概念（Key Concepts）
   - 涉及文件列表（Files）
   - 代码错误与修复（Errors）
   - 问题解决过程（Problem Solving）
   - 所有用户消息（User Messages）
   - 待办事项（Pending Tasks）
   - 当前工作（Current Work）
   - 下一步计划（Next Step）

**压缩后自动恢复文件上下文**：压缩会丢弃就消息，但会话期间所有被读过、写过、编辑过的文件都记录在 readFileState 缓存中（包括 agent 调用 FileRead/FileEdit/FileWrite/Bash 工具操作的文件，以及 memdir系统和 Sonnet预取自动加载的记忆文件）。压缩完成后，系统会从这个缓存中排除 [CLAUDE.md](http://CLAUDE.md) 和 Plan 文件（他们有各自的恢复机制），再按最近访问时间排序，取最多 5 个文件从磁盘重新读取（不用旧缓存，确保内容是最新的），以每文件≤5K tokens，总计 ≤50K tokens 的预算注入到压缩后的上下文中，让模型不需要在下一轮重新读取这些文件就能继续工作。如果会话中没有操作过任何文件，则不会恢复。

**防御机制**：连续失败3次后触发熔断，不再尝试。如果压缩调用本身也报 prompt-too-long，会从最旧的消息开始截断，最多重试 3 次

### 第四层：Blocking Limit（硬拒绝）

如果经过前三层压缩后 token 数仍然超过硬上限（有效窗口 - 3000，约177K），直接向用户输出 PROMPT_TOO_LONG_ERROR_MESSAGE 并终止本轮。这是发起 API 调用前的最后一道门。

### 第五层：Reactive Compact（事后兜底）

在 API 调用之后触—当API返回 prompt_too_long（413）或媒体文件过大错误时，拦截错误，对旧消息做一次摘要压缩，然后重试 API 调用。每轮最多尝试 1 次，防止死循环。

### 总结

**Q&A**

**Q：为什么建议主动 /compact**

**A：**自动压缩触发的时候，无法控制保留什么，系统按照它的模板做摘要，可能会丢掉你认为重要的上下文。而手动 /compact 可以附带指令，让 LLM 知道该重点保留什么。自动压缩是实现“本质上无限上下文”（essentially infinite context）的核心手段，但这更多的是系统兜底，用户主动管理上下文会更好。

## 记忆写入

三种写入策略

1. **extractMemories（持久记忆提取）**在每次完整查询结束后（模型给出最终回复），由一个 Fork Agent 后台执行：

   - 继承父对话的完整上下文，共享 prompt cache
   - **严格权限沙箱**：只能读文件 + 只能写记忆目录，不能执行bash、不能调 MCP、不能启动子代理
   - 高效的 2 轮模式：第 1 轮并行读取所有需更新的文件 →第 2 轮并行写所有变更
   - 有频率限制，不是每轮都跑
2. **Session Memory 更新**对话 token 超过 10K 后启动，之后每 5K tokens 由后台 Fork Agent 更新一次结构化笔记文件。
3. **AutoDream（记忆整合）—“睡眠整理“**定期运行的后台代理，对持久记忆做整理、合并、去重、清理。**触发条件（三个必须全通过才执行）：**

   - 距离上次整合≥ **24 小时**（最先检查，成本最低— 1 次 stat 调用）
   - 距离上次整合≥**5个会话**
   - 获取 **consolidation lock**（PID 文件锁，防并发；超过 1 小时且 PID 已死则可回收）
   - **Orient**：读取 [MEMORY.md](http://MEMORY.md) 索引，浏览现有主题文件，了解当前状态
   - **Gather**：扫描每日日志和会话记录（JSONL），窄范围 grep ，不穷举
   - **Consolidate**：新信息合并到现有主题文件，修复矛盾，相对日期转化为绝对日期
   - **Prune**：保持 [MEMORY.md](http://MEMORY.md) ≤ 200行，删除过时条目，精简冗长描述
